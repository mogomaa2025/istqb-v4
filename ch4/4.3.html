4.3.1 Statement Testing and Statement Coverage

"In statement testing, the coverage items are executable statements. The aim is to design test cases that exercise statements in the code until an acceptable level of coverage is achieved."

في Statement Testing، العناصر اللي بنقيس تغطيتها هي executable statements (البيانات القابلة للتنفيذ).

💻 مثال كود عملي مع Statement Coverage:

1:int calculateDiscount(int age, boolean isStudent) {
2: int discount = 0; // Statement 1
3: if (age >= 65) { // Statement 2
4: discount = 20; // Statement 3
5: }
6: if (isStudent) { // Statement 4
7: discount += 10; // Statement 5
8: }
9: return discount; // Statement 6
10:}

📊 Total Executable Statements: 6

📐

Statement Coverage Formula

"Coverage is measured as the number of statements exercised by the test cases divided by the total number of executable statements in the code, and is expressed as a percentage."
📐 معادلة Statement Coverage:

Statement Coverage = (عدد البيانات المختبرة ÷ العدد الكلي للبيانات القابلة للتنفيذ) × 100%

🔢 مثال حسابي تفصيلي:

Test Case Input Values Statements Executed Coverage
TC1 age=70, isStudent=true 1, 2, 3, 4, 5, 6 100%
TC2 age=25, isStudent=false 1, 2, 4, 6 67%
TC3 age=30, isStudent=true 1, 2, 4, 5, 6 83%
✅

100% Statement Coverage - What it Guarantees

"When 100% statement coverage is achieved, it ensures that all executable statements in the code have been exercised at least once. In particular, this means that each statement with a defect will be executed, which may cause a failure demonstrating the presence of the defect."

✅ ما يضمنه 100% Statement Coverage

  • • كل statement تم تشغيله مرة واحدة على الأقل
  • • أي statement فيه عيب سيتم تنفيذه
  • • قد يسبب failure يظهر وجود العيب
  • • تأكيد من تشغيل كل أجزاء الكود

⚠️ ما لا يضمنه Statement Coverage

  • • لا يكتشف كل العيوب
  • • لا يختبر كل المنطق
  • • قد يفوت عيوب معتمدة على البيانات
  • • لا يضمن اختبار كل التفرعات
🚨

Statement Coverage Limitations

"However, exercising a statement with a test case will not detect defects in all cases. For example, it may not detect defects that are data dependent (e.g., a division by zero that only fails when a denominator is set to zero). Also, 100% statement coverage does not ensure that all the decision logic has been tested as, for instance, it may not exercise all the branches (see chapter 4.3.2) in the code."

🐛 أمثلة على العيوب اللي Statement Coverage قد يفوتها:

1️⃣ Data Dependent Defects:
1:int result = x / y; // Division by zero defect
2:return result;
  • ✅ Test Case 1: x=10, y=2 → Works fine, statement covered
  • ❌ Test Case 2: x=10, y=0 → Division by zero error!
2️⃣ Missing Branch Logic:
1:if (x > 0) { // Decision point
2: result = "positive"; // TRUE branch
3:} else {
4: result = "negative"; // FALSE branch
5:}
6:return result;
  • ✅ Test with x=5: يشغل statements 1,2,6 → 100% Statement Coverage
  • ❌ لكن: الـ FALSE branch (statement 4) لم يتم اختباره!

4.3.2 Branch Testing and Branch Coverage

"A branch is a transfer of control between two nodes in the control flow graph, which shows the possible sequences in which source code statements are executed in the test object."

Branch هو نقل السيطرة بين نقطتين في control flow graph.

🌳

Branch Types

"Each transfer of control can be either unconditional (i.e., straight-line code) or conditional (i.e., a decision outcome)."

➡️ Unconditional Branch

Straight-line code

x = 5;
y = x + 1; // Always executes
z = y * 2;

تتنفذ دائماً بالتتابع

🤔 Conditional Branch

Decision outcome

if (x > 0) { // TRUE branch
y = 1;
} else { // FALSE branch
y = -1;
}

تعتمد على شرط معين

🎯

Branch Testing Definition and Coverage

"In branch testing the coverage items are branches and the aim is to design test cases to exercise branches in the code until an acceptable level of coverage is achieved. Coverage is measured as the number of branches exercised by the test cases divided by the total number of branches, and is expressed as a percentage."
📐 معادلة Branch Coverage:

Branch Coverage = (عدد التفرعات المختبرة ÷ العدد الكلي للتفرعات) × 100%

🌳 مثال شامل Branch Testing:

1:int processOrder(int quantity, boolean isPremium) {
2: int fee = 0;
3: if (quantity > 100) { // Branch 1: TRUE/FALSE
4: fee = 50; // TRUE branch
5: } else {
6: fee = 100; // FALSE branch
7: }
8: if (isPremium) { // Branch 2: TRUE/FALSE
9: fee = fee / 2; // TRUE branch
10: }
11: return fee; // Unconditional
12:}
Branch ID Type Path Test Required
B1-T Conditional quantity > 100 (TRUE) quantity = 150
B1-F Conditional quantity > 100 (FALSE) quantity = 50
B2-T Conditional isPremium (TRUE) isPremium = true
B2-F Conditional isPremium (FALSE) isPremium = false
U1 Unconditional return statement Any test case

🎯 Total Branches: 5 | Minimum Test Cases for 100%: 2

🏆

100% Branch Coverage

"When 100% branch coverage is achieved, all branches in the code, unconditional and conditional, are exercised by test cases. Conditional branches typically correspond to a true or false outcome from an "if...then" decision, an outcome from a switch/case statement, or a decision to exit or continue in a loop."

🎯 أمثلة Conditional Branches:

  • 🔀 if...then decision: نتيجة true أو false
  • 🎛️ switch/case statement: كل حالة في switch
  • 🔄 Loop decision: قرار الخروج أو الاستمرار في loop

Branch Coverage Subsumes Statement Coverage

"Branch coverage subsumes statement coverage. This means that any set of test cases achieving 100% branch coverage also achieves 100% statement coverage (but not vice versa)."

Branch Coverage

100%

⟹
Statement Coverage

100%

✅ صحيح دائماً

100% Branch Coverage → 100% Statement Coverage

إذا اختبرت كل التفرعات، تلقائياً تكون اختبرت كل البيانات

❌ ليس صحيحاً دائماً

100% Statement Coverage ↛ 100% Branch Coverage

يمكن اختبار كل البيانات دون اختبار كل التفرعات

⚠️

Branch Coverage Limitations

"However, exercising a branch with a test case will not detect defects in all cases. For example, it may not detect defects requiring the execution of a specific path in a code."

🐛 مثال: Path-dependent Defects

1:if (userType == "admin") {
2: accessLevel = "high";
3:}
4:if (accessLevel == "high") { // Bug: accessLevel might be undefined!
5: deleteAllData();
6:}
  • ✅ Branch Coverage: يختبر كل branch منفصل
  • ❌ Path Coverage: مطلوب لاكتشاف العيب في المسار الكامل
  • 🚨 العيب: إذا userType != "admin" و accessLevel غير معرف

4.3.3 The Value of White-box Testing

"A fundamental strength that all white-box techniques share is that the entire software implementation is taken into account during testing, which facilitates defect detection even when the software specification is vague, outdated or incomplete."

💪 نقاط القوة الأساسية

  • 🔍 Complete Implementation Review: فحص شامل لكامل التطبيق
  • 📋 Works with Poor Specs: يعمل مع مواصفات ضعيفة
  • ⏰ Works with Outdated Specs: يعمل مع مواصفات قديمة
  • 🔧 Works with Incomplete Specs: يعمل مع مواصفات ناقصة
  • 📊 Objective Measurement: قياس موضوعي للتغطية

⚠️ نقاط الضعف

  • 🚫 Defects of Omission: لا يكتشف المتطلبات المفقودة
  • 📝 Missing Features: لا يكتشف الميزات غير المطبقة
  • 🎯 Implementation Focused: يركز على الكود فقط
  • 👤 User Perspective Missing: قد يفقد منظور المستخدم
🚨

White-Box Weakness: Defects of Omission

"A corresponding weakness is that if the software does not implement one or more requirements, white box testing may not detect the resulting defects of omission."

🐛 مثال: Defects of Omission

📋 المطلوب في المواصفات:
  • • نظام login يتطلب username + password
  • • تشفير كلمة المرور
  • • تسجيل محاولات الدخول الفاشلة
  • • قفل الحساب بعد 3 محاولات خاطئة
💻 المنفذ في الكود:
if (username == "admin" && password == "123") {
return "Login Successful";
} else {
return "Login Failed";
}
🚫 نتيجة White-Box Testing:

✅ White-Box هيختبر الكود الموجود ويحقق 100% coverage

❌ لكن مش هيكتشف إن التشفير وتسجيل المحاولات والقفل مفقودين!

📝

White-Box in Static Testing

"White-box techniques can be used in static testing (e.g., during dry runs of code). They are well suited to reviewing code that is not yet ready for execution, as well as pseudocode and other high-level or top-down logic which can be modeled with a control flow graph."

🔧 استخدامات White-Box في Static Testing:

  • 📝 Code Review: مراجعة كود مش جاهز للتنفيذ
  • 🧩 Pseudocode Analysis: تحليل الكود الوصفي
  • 📊 High-level Logic Review: مراجعة المنطق عالي المستوى
  • 🗺️ Control Flow Modeling: نمذجة مسار التحكم
  • 🔍 Dry Run Analysis: تحليل التشغيل الجاف

📊 مثال: Pseudocode Analysis

PSEUDOCODE:
BEGIN ProcessPayment
  INPUT: amount, cardNumber, cvv
  IF amount > 0 AND amount <= 10000 THEN
    IF ValidateCard(cardNumber, cvv) THEN
      ChargeCard(amount)
      RETURN "Success"
    ELSE
      RETURN "Invalid Card"
    END IF
  ELSE
    RETURN "Invalid Amount"
  END IF
END ProcessPayment

🎯 يمكن تطبيق Branch Coverage على هذا Pseudocode قبل البرمجة!

📊

Objective Coverage Measurement

"Performing only black-box testing does not provide a measure of actual code coverage. White-box coverage measures provide an objective measurement of coverage and provide the necessary information to allow additional tests to be generated to increase this coverage, and subsequently increase confidence in the code."

❌ Black-Box Only

  • • لا يقيس تغطية الكود الفعلية
  • • لا نعرف أي جزء من الكود لم يختبر
  • • صعوبة في تحديد اختبارات إضافية
  • • اعتماد على خبرة المختبر فقط

✅ White-Box Coverage

  • • قياس موضوعي ودقيق للتغطية
  • • معرفة تحديداً ما لم يختبر
  • • معلومات لإنتاج اختبارات إضافية
  • • زيادة الثقة في الكود بشكل مقيس

🎯 الحل المثالي

Black-Box + White-Box = Comprehensive Testing

استخدام White-Box لقياس التغطية و Black-Box لضمان متطلبات المستخدم
