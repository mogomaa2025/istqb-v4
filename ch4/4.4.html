4.4.1 Error Guessing

"Error guessing is a technique used to anticipate the occurrence of errors, defects, and failures, based on the tester's knowledge, including: How the application has worked in the past, The types of errors the developers tend to make and the types of defects that result from these errors, The types of failures that have occurred in other, similar applications."

Error Guessing هي تقنية لتوقع حدوث الأخطاء والعيوب والإخفاقات بناءً على معرفة المختبر.

🎯 مصادر المعرفة في Error Guessing

Past Application Behavior

Developer Error Patterns
أنماط أخطاء المطورين المتكررة
Similar Applications Failures
إخفاقات التطبيقات المشابهة
🐛

Error Categories

"In general, errors, defects and failures may be related to: input (e.g., correct input not accepted, parameters wrong or missing), output (e.g., wrong format, wrong result), logic (e.g., missing cases, wrong operator), computation (e.g., incorrect operand, wrong computation), interfaces (e.g., parameter mismatch, incompatible types), or data (e.g., incorrect initialization, wrong type)."

🗂️ فئات الأخطاء الشائعة

📥 Input Errors
  • • مدخل صحيح مرفوض
  • • معاملات خاطئة أو مفقودة
  • • تحقق غير صحيح من المدخلات
📤 Output Errors
  • • تنسيق خاطئ للمخرجات
  • • نتائج غير صحيحة
  • • رسائل خطأ غير واضحة
🧠 Logic Errors
  • • حالات مفقودة
  • • استخدام عوامل خاطئة
  • • منطق شرطي خاطئ
🔢 Computation Errors
  • • معامل غير صحيح
  • • حسابات خاطئة
  • • أخطاء في التقريب
🔌 Interface Errors
  • • عدم تطابق المعاملات
  • • أنواع غير متوافقة
  • • مشاكل في الاتصال
💾 Data Errors
  • • تهيئة غير صحيحة
  • • نوع بيانات خاطئ
  • • فقدان أو تلف البيانات

Fault Attacks

"Fault attacks are a methodical approach to the implementation of error guessing. This technique requires the tester to create or acquire a list of possible errors, defects and failures, and to design tests that will identify defects associated with the errors, expose the defects, or cause the failures."

Fault Attacks هو النهج المنهجي لتطبيق Error Guessing.

4.4.2 Exploratory Testing

"In exploratory testing, tests are simultaneously designed, executed, and evaluated while the tester learns about the test object. The testing is used to learn more about the test object, to explore it more deeply with focused tests, and to create tests for untested areas."

في Exploratory Testing، الاختبارات يتم تصميمها وتنفيذها وتقييمها في نفس الوقت بينما يتعلم المختبر عن الكائن المختبر.

Session-based Testing

"Exploratory testing is sometimes conducted using session-based testing to structure the testing. In a session-based approach, exploratory testing is conducted within a defined time-box. The tester uses a test charter containing test objectives to guide the testing."

📅 مكونات Session-based Testing

⏰ Time-box

إطار زمني محدد للجلسة

  • • عادة 60-120 دقيقة
  • • تركيز مكثف
  • • لا مقاطعات

🎯 Test Charter

ميثاق يحتوي على أهداف الاختبار

  • • أهداف واضحة
  • • نطاق محدد
  • • إرشادات للمختبر

📋 Session Sheets

توثيق الخطوات والاكتشافات

  • • الخطوات المتبعة
  • • النتائج المكتشفة
  • • المشاكل المحتملة

🗣️ Debriefing

  • • مراجعة النتائج
  • • تحديد الخطوات التالية
  • • تحديث الاستراتيجية

When to Use Exploratory Testing

"Exploratory testing is useful when there are few or inadequate specifications or there is significant time pressure on the testing. Exploratory testing is also useful to complement other more formal test techniques."
Situation Why Exploratory Testing? Benefits Example
Few Specifications لا توجد مواصفات كافية اكتشاف سلوك غير موثق تطبيق جديد في بيتا
Inadequate Specs المواصفات غير مكتملة ملء الفجوات في المتطلبات مواصفات قديمة أو غامضة
Time Pressure ضغط زمني شديد اختبار سريع وفعال إصدار طارئ أو hotfix
Complement Formal تكملة للاختبارات الرسمية كشف عيوب فاتت التقنيات الأخرى بعد تنفيذ test cases رسمية

4.4.3 Checklist-Based Testing

"In checklist-based testing, a tester designs, implements, and executes tests to cover test conditions from a checklist. Checklists can be built based on experience, knowledge about what is important for the user, or an understanding of why and how software fails."

في Checklist-based Testing، المختبر يصمم وينفذ ويؤدي الاختبارات لتغطية شروط الاختبار من قائمة مراجعة.

📋 مصادر بناء Checklists

Experience

Checklist Guidelines

"Checklists should not contain items that can be checked automatically, items better suited as entry/exit criteria, or items that are too general. Checklist items are often phrased in the form of a question. It should be possible to check each item separately and directly."

✅ ما يجب تضمينه

  • • أسئلة واضحة ومحددة
  • • عناصر قابلة للفحص منفرداً
  • • متطلبات واجهة المستخدم
  • • خصائص الجودة
  • • شروط الاختبار الواضحة

❌ ما يجب تجنبه

  • • عناصر قابلة للفحص تلقائياً
  • • معايير الدخول/الخروج
  • • عناصر عامة جداً
  • • أسئلة غامضة
  • • عناصر معقدة مترابطة

✓

✓

Checklist Types and Applications

"These items may refer to requirements, graphical interface properties, quality characteristics or other forms of test conditions. Checklists can be created to support various test types, including functional and non-functional testing (e.g., 10 heuristics for usability testing (Nielsen 1994))."
Checklist Type Focus Area Example Items Usage Context
Requirements تحقق من المتطلبات هل تم تنفيذ كل الوظائف المطلوبة؟ Functional Testing
GUI Properties خصائص واجهة المستخدم هل الألوان متسقة؟ هل النصوص واضحة؟ UI/UX Testing
Quality Characteristics خصائص الجودة هل الاستجابة سريعة؟ هل النظام آمن؟ Non-functional Testing
Usability Heuristics سهولة الاستخدام Nielsen's 10 Usability Heuristics Usability Testing
Security الأمان والحماية هل المصادقة قوية؟ هل البيانات محمية؟ Security Testing
Performance الأداء والسرعة هل زمن التحميل مقبول؟ هل يتعامل مع الحمل؟ Performance Testing

Checklist Maintenance

"Some checklist entries may gradually become less effective over time because the developers will learn to avoid making the same errors. New entries may also need to be added to reflect newly found high severity defects. Therefore, checklists should be regularly updated based on defect analysis."

Benefits and Limitations

"In the absence of detailed test cases, checklist-based testing can provide guidelines and some degree of consistency for the testing. If the checklists are high-level, some variability in the actual testing is likely to occur, resulting in potentially greater coverage but less repeatability."

✅ المميزات

  • • توفر إرشادات في غياب test cases مفصلة
  • • تحقق درجة من الاتساق في الاختبار
  • • مرونة في التنفيذ
  • • تغطية أوسع محتملة
  • • سهولة في الاستخدام والتطبيق

⚠️ القيود

  • • قابلية تكرار أقل
  • • تباين في التنفيذ الفعلي
  • • تعتمد على خبرة المختبر
  • • قد تفقد فعاليتها مع الوقت
  • • تحتاج صيانة منتظمة
