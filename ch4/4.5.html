4.5.1 Collaborative User Story Writing

"A user story represents a feature that will be valuable to either a user or purchaser of a system or software. User stories have three critical aspects, called together the "3 C's": Card – the medium describing a user story, Conversation – explains how the software will be used, Confirmation – the acceptance criteria."

User Story تمثل ميزة ستكون قيمة للمستخدم أو مشتري النظام. لها 3 جوانب حرجة تسمى "3 C's":

📋 الـ 3 C's في User Stories

📄
Card
الوسيط الوصفي
💬
Conversation
شرح الاستخدام
✅
Confirmation
معايير القبول
📝

User Story Format

"The most common format for a user story is "As a [role], I want [goal to be accomplished], so that I can [resulting business value for the role]", followed by the acceptance criteria."

📋 تنسيق User Story القياسي

📐 Template:
As a [role],
I want [goal to be accomplished],
so that I can [resulting business value for the role]
🔍 أمثلة عملية:
مثال 1: تطبيق مصرفي
As a bank customer,
I want to transfer money between my accounts,
so that I can manage my finances efficiently without visiting the branch.
مثال 2: متجر إلكتروني
As a online shopper,
I want to save items to a wishlist,
so that I can purchase them later when convenient.
🤝

Collaborative Authorship

"Collaborative authorship of the user story can use techniques such as brainstorming and mind mapping. The collaboration allows the team to obtain a shared vision of what should be delivered, by taking into account three perspectives: business, development and testing."

🎯 ثلاث منظورات في التعاون

Shared Vision
رؤية مشتركة
Business
المتطلبات والقيمة
Development
التنفيذ والتقنية
Testing
الجودة والتحقق

🧠 تقنيات التعاون في كتابة User Stories:

  • 🌟 Brainstorming: جلسات عصف ذهني لتوليد الأفكار
  • 🗺️ Mind Mapping: خرائط ذهنية لتنظيم المفاهيم
  • 👥 Cross-functional Teams: فرق متعددة التخصصات
  • 🎯 Story Workshops: ورش عمل متخصصة
  • 📊 Story Mapping: خرائط الـ User Stories
💎

INVEST Criteria

"Good user stories should be: Independent, Negotiable, Valuable, Estimable, Small and Testable (INVEST). If a stakeholder does not know how to test a user story, this may indicate that the user story is not clear enough, or that it does not reflect something valuable to them, or that the stakeholder just needs help in testing."

💎 معايير INVEST لـ User Stories الجيدة

I - Independent
مستقلة عن user stories أخرى
N - Negotiable
قابلة للتفاوض والتعديل
V - Valuable
ذات قيمة للمستخدم أو العمل
E - Estimable
قابلة للتقدير والحساب
S - Small
صغيرة ومحددة النطاق
T - Testable
قابلة للاختبار والتحقق
⚠️ علامات تحذيرية:

إذا لم يعرف صاحب المصلحة كيفية اختبار user story، قد يعني هذا:

  • 📋 الـ user story غير واضحة بما فيه الكفاية
  • 💰 لا تعكس شيئاً قيماً لهم
  • 🤝 يحتاج صاحب المصلحة للمساعدة في الاختبار

4.5.2 Acceptance Criteria

"Acceptance criteria for a user story are the conditions that an implementation of the user story must meet to be accepted by stakeholders. From this perspective, acceptance criteria may be viewed as the test conditions that should be exercised by the tests."

Acceptance Criteria هي الشروط التي يجب أن يلبيها تنفيذ الـ user story ليتم قبوله من أصحاب المصلحة. يمكن اعتبارها test conditions التي يجب اختبارها.

🎯

Uses of Acceptance Criteria

"Acceptance criteria are used to: Define the scope of the user story, Reach consensus among the stakeholders, Describe both positive and negative scenarios, Serve as a basis for the user story acceptance testing, Allow accurate planning and estimation."
الاستخدام الوصف الفائدة مثال
تحديد النطاق Define scope of user story وضوح في الحدود ما يجب تنفيذه وما لا يجب
إجماع أصحاب المصلحة Reach consensus among stakeholders تفاهم مشترك اتفاق على التوقعات
السيناريوهات الإيجابية والسلبية Positive and negative scenarios تغطية شاملة الحالات العادية والاستثنائية
أساس اختبار القبول Basis for acceptance testing معايير واضحة للاختبار تحويل المعايير لـ test cases
التخطيط والتقدير Planning and estimation تقديرات دقيقة حساب الجهد والوقت المطلوب
📋

Acceptance Criteria Formats

"There are several ways to write acceptance criteria for a user story. The two most common formats are: Scenario-oriented (e.g., Given/When/Then format used in BDD), Rule-oriented (e.g., bullet point verification list, or tabulated form of input-output mapping)."

📊 مقارنة أنواع تنسيقات Acceptance Criteria

🎬 Scenario-oriented

Given/When/Then Format (BDD)
Given
a registered user is logged in

When
they click on "Transfer Money"

Then
the transfer form should appear
مميزات: واضح للجميع، يشبه اللغة الطبيعية

📝 Rule-oriented

Bullet Points / Tabulated Form
  • ✅ User must be authenticated
  • ✅ Transfer amount > 0
  • ✅ Sufficient balance required
  • ✅ Valid destination account
  • ❌ Transfer limits enforced
مميزات: مباشر ومفصل، سهل التحقق

💡 مرونة في التنسيق

يمكن للفريق استخدام تنسيق مخصص طالما أن acceptance criteria واضحة وغير غامضة

4.5.3 Acceptance Test-driven Development (ATDD)

"ATDD is a test-first approach. Test cases are created prior to implementing the user story. The test cases are created by team members with different perspectives, e.g., customers, developers, and testers."

ATDD هو نهج test-first حيث يتم إنشاء test cases قبل تنفيذ الـ user story بواسطة أعضاء الفريق من منظورات مختلفة.

👥 فريق ATDD متعدد المنظورات

ATDD Team
فريق متعدد التخصصات
Customers
منظور المستخدم
Developers
منظور التقني
Testers
منظور الجودة
🔄

ATDD Workflow

"The first step is a specification workshop where the user story and (if not yet defined) its acceptance criteria are analyzed, discussed, and written by the team members. The next step is to create the test cases."

🔄 خطوات ATDD Workflow

🏛️
Specification Workshop
تحليل ومناقشة User Story
📝
Define Acceptance Criteria
تحديد معايير القبول
🧪
Create Test Cases
إنشاء حالات الاختبار
💻
Implement Code
تنفيذ الكود
✅
Run Tests
تشغيل الاختبارات

🎯 فوائد Specification Workshop:

  • 🔍 حل عدم الاكتمال: معالجة النقص في الـ user story
  • 💡 إزالة الغموض: توضيح النقاط المبهمة
  • 🐛 إصلاح العيوب: تحديد وحل العيوب المبكرة
  • 🤝 تفاهم مشترك: ضمان فهم موحد بين الفريق
🧪

ATDD Test Case Characteristics

"Test cases are based on the acceptance criteria and can be seen as examples of how the software works. Since examples and tests are the same, these terms are often used interchangeably."

حالات الاختبار في ATDD مبنية على acceptance criteria ويمكن اعتبارها أمثلة على كيفية عمل البرنامج.

🔄 Examples = Tests

Examples ⟷ Tests

المصطلحان يُستخدمان بالتبادل في ATDD

📊

ATDD Test Sequence

"Typically, the first test cases are positive, confirming the correct behavior without exceptions or error conditions. After the positive test cases are done, the team should perform negative testing. Finally, the team should cover non-functional quality characteristics as well."

📊 تسلسل اختبارات ATDD

1
Positive Test Cases
تأكيد السلوك الصحيح بدون استثناءات أو حالات خطأ
🎯 المسار السعيد (Happy Path)
2
Negative Testing
اختبار حالات الخطأ والاستثناءات
🚨 حالات الأخطاء والحدود
3
Non-functional Testing
تغطية خصائص الجودة غير الوظيفية
⚡ الأداء، قابلية الاستخدام، الأمان
📝

ATDD Test Case Requirements

"Test cases should be expressed in a way that is understandable for the stakeholders. Test cases must cover all the characteristics of the user story and should not go beyond the story. No two test cases should describe the same characteristics of the user story."

📋 متطلبات Test Cases في ATDD:

  • 👥 مفهومة للجميع: لغة طبيعية مفهومة لأصحاب المصلحة
  • 📦 تغطية شاملة: تغطي كل خصائص الـ user story
  • 🚫 عدم تجاوز النطاق: لا تتجاوز حدود الـ story
  • 🔄 عدم التكرار: لا يوجد test case يصف نفس الخصائص
  • 📋 العناصر الأساسية: preconditions + inputs + postconditions

📋 مثال: ATDD Test Case Structure

🏦 User Story: Money Transfer

"As a bank customer, I want to transfer money..."

✅ Test Case 1: Successful Transfer
Precondition: User logged in, balance ≥ 1000
Input: Transfer 500 to valid account
Expected Result: Transfer successful, balance reduced by 500
❌ Test Case 2: Insufficient Balance
Precondition: User logged in, balance = 100
Input: Transfer 500 to valid account
Expected Result: Error message "Insufficient funds"
⚙️

Executable Requirements

"When captured in a format supported by a test automation framework, the developers can automate the test cases by writing the supporting code as they implement the feature described by a user story. The acceptance tests then become executable requirements."

عندما يتم التقاط test cases بتنسيق مدعوم من إطار أتمتة الاختبار، يصبح اختبار القبول executable requirements.

⚙️ من Test Cases إلى Executable Requirements

📝
Write Test Cases
كتابة حالات الاختبار
🔧
Automation Framework
إطار الأتمتة
💻
Supporting Code
كتابة الكود المساعد
⚙️
Executable Requirements
متطلبات قابلة للتنفيذ

🏆 فوائد Executable Requirements:

  • 🤖 أتمتة كاملة: تشغيل الاختبارات تلقائياً
  • 🔄 تحديث مستمر: تطوير مدفوع بالاختبارات
  • 📋 وثائق حية: المتطلبات تُختبر فعلياً
  • ✅ تحقق فوري: تأكيد أن الكود يلبي المتطلبات
  • 🚀 تطوير أسرع: feedback فوري أثناء التطوير

Integration with Other Test Techniques

"During the test design the test techniques described in sections 4.2, 4.3 and 4.4 may be applied."

أثناء تصميم الاختبار في ATDD، يمكن تطبيق تقنيات الاختبار المذكورة في الأقسام 4.2، 4.3، و 4.4.

Test Technique Category Specific Techniques Application in ATDD Example
