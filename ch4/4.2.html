4.2.1 Equivalence Partitioning (EP)

"Equivalence Partitioning (EP) divides data into partitions (known as equivalence partitions) based on the expectation that all the elements of a given partition are to be processed in the same way by the test object."

Equivalence Partitioning بيقسم البيانات إلى مجموعات تسمى equivalence partitions بناءً على توقع إن كل العناصر في المجموعة الواحدة هتتعامل معاها البرنامج بنفس الطريقة.

📊 مثال بصري: تقسيم الأعمار

< 18
Invalid
18 - 65
Valid
> 65
Invalid

🎯 اختبار واحد لكل مجموعة: 10 سنة، 30 سنة، 70 سنة

💡

EP Theory and Data Types

"The theory behind this technique is that if a test case, that tests one value from an equivalence partition, detects a defect, this defect should also be detected by test cases that test any other value from the same partition. Therefore, one test for each partition is sufficient."

🧠 النظرية: لو test case اختبر قيمة من مجموعة ولقى عيب، أي قيمة تانية من نفس المجموعة هتلاقي نفس العيب.

نوع البيانات أمثلة خصائص التقسيم
Inputs User input fields, parameters Valid/Invalid ranges
Outputs System responses, results Expected/Error outputs
Configuration Items Settings, preferences Supported/Unsupported configs
Time-related Values Dates, timestamps Past/Present/Future
Interface Parameters API calls, data formats Compatible/Incompatible
🔍

Partition Properties

"Equivalence partitions can be identified for any data element related to the test object, including inputs, outputs, configuration items, internal values, time-related values, and interface parameters. The partitions may be continuous or discrete, ordered or unordered, finite or infinite. The partitions must not overlap and must be non-empty sets."

📊 Continuous

أرقام متصلة (1.5، 2.7، 3.9)

🔢 Discrete

أرقام منفصلة (1، 2، 3)

📈 Ordered

لها ترتيب (صغير → كبير)

🎲 Unordered

بدون ترتيب (ألوان، أسماء)

⚠️ قواعد مهمة للمجموعات:

  • 🚫 Must not overlap: لا تتداخل المجموعات
  • 📋 Non-empty sets: كل مجموعة تحتوي على عناصر
  • 🎯 Complete coverage: تغطي كل الحالات الممكنة
📊

EP Coverage Measurement

"In EP, the coverage items are the equivalence partitions. To achieve 100% coverage with this technique, test cases must exercise all identified partitions (including invalid partitions) by covering each partition at least once. Coverage is measured as the number of partitions exercised by at least one test case, divided by the total number of identified partitions, and is expressed as a percentage."
📐 معادلة EP Coverage:

EP Coverage = (عدد المجموعات المختبرة ÷ العدد الكلي للمجموعات) × 100%

🔢 مثال حسابي شامل:

المجموعة النوع القيم تم الاختبار؟
Partition 1 Invalid < 0 ✅
Partition 2 Valid 0 - 100 ✅
Partition 3 Invalid > 100 ❌

EP Coverage = (2 ÷ 3) × 100% = 66.7%

🎯

Each Choice Coverage

"Many test objects include multiple sets of partitions (e.g., test objects with more than one input parameter), which means that a test case will cover partitions from different sets of partitions. The simplest coverage criterion in the case of multiple sets of partitions is called Each Choice coverage. Each Choice coverage requires test cases to exercise each partition from each set of partitions at least once. Each Choice coverage does not take into account combinations of partitions."

Each Choice Coverage بيستخدم مع الأنظمة اللي عندها أكتر من مجموعة partitions.

🎮 مثال: نظام تسجيل لعبة

Parameter Partition 1 Partition 2 Partition 3
Age < 13 (Invalid) 13-17 (Valid) 18+ (Valid)
Country Supported Restricted -
🎯 Each Choice Coverage:

نحتاج لاختبار كل partition مرة واحدة على الأقل، مش كل التركيبات الممكنة

Test Cases needed: 3 بدلاً من 6 (3×2) تركيبات

4.2.2 Boundary Value Analysis (BVA)

"Boundary Value Analysis (BVA) is a technique based on exercising the boundaries of equivalence partitions. Therefore, BVA can only be used for ordered partitions."

Boundary Value Analysis بتركز على اختبار الحدود بين المجموعات.

Boundary Definition and Properties

"The minimum and maximum values of a partition are its boundary values. In the case of BVA, if two elements belong to the same partition, all elements between them must also belong to that partition."

القيم الدنيا والعليا للمجموعة هي boundary values، وأي عناصر بينهما لازم تنتمي لنفس المجموعة.

🎯 لماذا BVA مهمة؟

  • 💻 Developer Errors: المطورين أكثر عرضة للخطأ في الحدود
  • 🐛 Common Mistakes: استخدام > بدل >= أو < بدل <=
  • 📍 Misplaced Boundaries: وضع الحدود في مكان خطأ
  • 🚫 Omitted Boundaries: نسيان الحدود تماماً
🔢

BVA Versions: 2-Value vs 3-Value

"This syllabus covers two versions of the BVA: 2-value and 3-value BVA. They differ in terms of coverage items per boundary that need to be exercised to achieve 100% coverage."
BVA Version Coverage Items per Boundary Example (Boundary = 10) Rigor Level
2-Value BVA Boundary + Closest Neighbor 10, 11 ⭐⭐⭐
3-Value BVA Boundary + Both Neighbors 9, 10, 11 ⭐⭐⭐⭐⭐

🐛 لماذا 3-Value أفضل؟

// خطأ في الكود الأصلي
if (x <= 10) { // المطلوب
if (x == 10) { // المنفذ بالخطأ

// 2-Value BVA: x=10, x=11 → لن يكتشف الخطأ
// 3-Value BVA: x=9, x=10, x=11 → سيكتشف الخطأ عند x=9

4.2.3 Decision Table Testing

"Decision tables are used for testing the implementation of system requirements that specify how different combinations of conditions result in different outcomes. Decision tables are an effective way of recording complex logic, such as business rules."

Decision Tables تستخدم لاختبار تنفيذ متطلبات النظام اللي بتحدد إزاي التركيبات المختلفة من الشروط بتؤدي لنتائج مختلفة.

🏦 مثال متقدم: نظام الموافقة على القرض

Conditions/Actions Rule 1 Rule 2 Rule 3 Rule 4 Rule 5
Credit Score ≥ 700 T T F F F
Income ≥ 50,000 T F T F –
Employment ≥ 2 years T T F T F
Approve Loan X X – – –
Manual Review – – X X –
Reject Loan – – – – X
🏗️

Decision Table Structure and Notation

"When creating decision tables, the conditions and the resulting actions of the system are defined. These form the rows of the table. Each column corresponds to a decision rule that defines a unique combination of conditions, along with the associated actions."

📏 الصفوف (Rows)

  • • Conditions (الشروط)
  • • Actions (الإجراءات)

📐 الأعمدة (Columns)

  • • Decision Rules
  • • Unique Combinations
Symbol Conditions Actions Meaning
T True - الشرط متحقق - Condition is satisfied
F False - الشرط غير متحقق - Condition is not satisfied
– Irrelevant - غير مهم - Value doesn't affect outcome
N/A Infeasible - غير ممكن - Condition is infeasible
X - Action occurs - الإجراء يحدث Action should occur
Blank - No action - لا إجراء Action should not occur
📊

Decision Table Coverage and Strengths

"In decision table testing, the coverage items are the columns containing feasible combinations of conditions. To achieve 100% coverage with this technique, test cases must exercise all these columns. Coverage is measured as the number of exercised columns, divided by the total number of feasible columns, and is expressed as a percentage."
📐 معادلة Decision Table Coverage:

Coverage = (عدد الأعمدة المختبرة ÷ العدد الكلي للأعمدة الممكنة) × 100%

💪 نقاط قوة Decision Table Testing:

  • 🎯 Systematic Approach: نهج منظم لتحديد كل التركيبات
  • 🔍 Gap Detection: يساعد في العثور على الثغرات في المتطلبات
  • 🚫 Contradiction Finding: يكشف التناقضات في المتطلبات
  • 📋 Complete Coverage: ضمان تغطية كل الحالات
⚠️ تحديات Decision Table:

Exponential Growth: عدد القواعد ينمو بشكل أُسي مع عدد الشروط

الحلول: استخدام Minimized Decision Table أو Risk-based Approach

Transition Syntax and State Table

"The common transition labeling syntax is as follows: "event [guard condition] / action". Guard conditions and actions can be omitted if they do not exist or are irrelevant for the tester."

State Transition Coverage Criteria

"There exist many coverage criteria for state transition testing. This syllabus discusses three of them: All States Coverage, Valid Transitions Coverage (also called 0-switch coverage), All Transitions Coverage."

📊 معايير التغطية الثلاثة

🏠 All States Coverage

الأضعف: زيارة كل الحالات

Coverage = (States Visited ÷ Total States) × 100%

يمكن تحقيقه بدون تشغيل كل الانتقالات

✅ Valid Transitions Coverage

الأكثر استخداماً: كل الانتقالات الصالحة

Coverage = (Valid Transitions Exercised ÷ Total Valid Transitions) × 100%

يضمن تحقيق All States Coverage

🔴 All Transitions Coverage

الأقوى: كل الانتقالات (صالحة + غير صالحة)

Coverage = (All Transitions Tested ÷ Total Transitions) × 100%

مطلوب للأنظمة الحساسة والحيوية

🏆 العلاقة بين معايير التغطية:
  • 🔗 All Transitions Coverage يضمن Valid Transitions Coverage
  • 🔗 Valid Transitions Coverage يضمن All States Coverage
  • ⚠️ All Transitions Coverage هو الحد الأدنى للأنظمة الحيوية
🎯

Coverage Relationship and Requirements

"All states coverage is weaker than valid transitions coverage, because it can typically be achieved without exercising all the transitions. Valid transitions coverage is the most widely used coverage criterion. Achieving full valid transitions coverage guarantees full all states coverage. Achieving full all transitions coverage guarantees both full all states coverage and full valid transitions coverage and should be a minimum requirement for mission and safety-critical software."

📈 Coverage Strength Comparison

All States
40% Strength
Valid Transitions
75% Strength
All Transitions
100% Strength

🚀 متى نستخدم كل معيار؟

  • 🏠 All States: للأنظمة البسيطة والتطبيقات العادية
  • ✅ Valid Transitions: للأنظمة التجارية والتطبيقات المهمة
  • 🔴 All Transitions: للأنظمة الطبية والطيران والسلامة
