4.1 Overview of Test Techniques

Learning Objective FL-4.1.1 (K2): Distinguish black-box, white-box and experience-based test techniques

What Are Test Techniques?

Core Philosophy: "العبرة بالكيف مش بالكم" (Quality over Quantity)
Create fewer but more effective and impactful tests
Primary Goals Description Benefit
Quality over Quantity نعمل اختبارات قليلة بس مفيدة ومؤثرة More effective testing with less effort
Systematic Coverage تغطية منهجية شاملة Ensure comprehensive testing without redundancy

Three Critical Elements Test Techniques Help Determine

Element Arabic Description Impact on Testing
Test Conditions الشروط والحالات اللي هنختبرها What scenarios to test
Coverage Items العناصر اللي لازم نغطيها في الاختبار What components need testing
Test Data البيانات المناسبة واللازمة للاختبار What inputs to use

The Three Main Categories

Technique Type Arabic Name Key Characteristic When to Use
Black-Box اختبار من الخارج Independent of implementation Test functionality without knowing internal structure
White-Box اختبار من الداخل Dependent on program design Test internal structure and code paths
Experience-Based بناءً على الخبرة Based on tester knowledge and intuition Complement formal techniques with expertise
Real-Life Example: Netflix streaming service
  • Black-box: Testing if movies play correctly regardless of how the streaming algorithm works internally
  • White-box: Testing the recommendation algorithm's code paths and decision logic
  • Experience-based: Using knowledge of common user behaviors to test edge cases like binge-watching or network interruptions
Exam Key Point: The Complete Testing Equation
Black-Box + White-Box + Experience-Based = Complete Testing

4.2.1 Equivalence Partitioning (EP)

Core Theory: If a test case from one partition finds a defect, any other value from the same partition should find the same defect.
Data Type Examples Partition Characteristics
Inputs User input fields, parameters Valid/Invalid ranges
Outputs System responses, results Expected/Error outputs
Configuration Items Settings, preferences Supported/Unsupported configs
Time-related Values Dates, timestamps Past/Present/Future
Interface Parameters API calls, data formats Compatible/Incompatible
EP Coverage = (Number of Partitions Tested ÷ Total Number of Partitions) × 100%
Real-Life Example: Amazon Prime Video Age Verification
  • Partition 1: Age < 13 (Invalid - Cannot create account)
  • Partition 2: Age 13-17 (Valid - Restricted content)
  • Partition 3: Age 18+ (Valid - Full access)
Test values: 10, 15, 25 (one from each partition)

Each Choice Coverage

Parameter Partition 1 Partition 2 Partition 3
Age < 13 (Invalid) 13-17 (Valid) 18+ (Valid)
Country Supported Restricted -
Efficiency Note: Need to test each partition at least once, not all possible combinations.
Test Cases needed: 3 instead of 6 (3×2) combinations

4.2.2 Boundary Value Analysis (BVA)

Focus: Test the boundaries between partitions where defects are most likely to occur.
BVA Version Coverage Items per Boundary Example (Boundary = 10) Rigor Level
2-Value BVA Boundary + Closest Neighbor 10, 11 ⭐⭐⭐
3-Value BVA Boundary + Both Neighbors 9, 10, 11 ⭐⭐⭐⭐⭐
Real-Life Example: PayPal Money Transfer Limits
  • Boundary: $10,000 daily limit
  • BVA Test Values: $9,999.99, $10,000.00, $10,000.01
  • Expected Results: Accept, Accept, Reject

4.2.3 Decision Table Testing

Purpose: Test how different combinations of conditions lead to different outcomes in business rules.

Decision Table Symbols

Symbol For Conditions For Actions Meaning
T True - الشرط متحقق - Condition is satisfied
F False - الشرط غير متحقق - Condition is not satisfied
– Irrelevant - غير مهم - Value doesn't affect outcome
N/A Infeasible - غير ممكن - Condition is infeasible
X - Action occurs - الإجراء يحدث Action should occur
Blank - No action - لا إجراء Action should not occur
Real-Life Example: Bank Loan Approval System
Conditions/Actions Rule 1 Rule 2 Rule 3 Rule 4 Rule 5
Credit Score ≥ 700 T T F F F
Income ≥ $50,000 T F T F –
Employment ≥ 2 years T T F T F
Approve Loan X X – – –
Manual Review – – X X –
Reject Loan – – – – X
Decision Table Coverage = (Columns Tested ÷ Total Feasible Columns) × 100%
Challenge: Exponential Growth - Number of rules grows exponentially with conditions
Solutions: Use Minimized Decision Table or Risk-based Approach

4.2.4 State Transition Testing

Purpose: Model system behavior by showing possible states and valid transitions between them. Format: event [guard condition] / action
Real-Life Example: ATM Machine States
Current State Event: insert_card Event: enter_pin Event: select_option
Idle Card Inserted / show_pin ❌ Invalid ❌ Invalid
Card Inserted ❌ Invalid PIN Verification / check_pin ❌ Invalid
Main Menu ❌ Invalid ❌ Invalid Transaction / start_process

State Transition Coverage Levels

Coverage Type Description Coverage Formula Strength
All States الأضعف: زيارة كل الحالات (States Visited ÷ Total States) × 100% ⭐⭐
Valid Transitions الأكثر استخداماً: كل الانتقالات الصالحة (Valid Transitions Exercised ÷ Total Valid Transitions) × 100% ⭐⭐⭐⭐
All Transitions الأقوى: كل الانتقالات (صالحة + غير صالحة) (All Transitions Tested ÷ Total Transitions) × 100% ⭐⭐⭐⭐⭐

4.3.1 Statement Testing

Statement Coverage = (Statements Executed ÷ Total Executable Statements) × 100%
Real-Life Example: E-commerce Discount Calculator
Test Case Input Values Statements Executed Coverage
TC1 age=70, isStudent=true 1, 2, 3, 4, 5, 6 100%
TC2 age=25, isStudent=false 1, 2, 4, 6 67%
TC3 age=30, isStudent=true 1, 2, 4, 5, 6 83%

4.3.2 Branch Testing

Branch Definition: Transfer of control between two points in control flow graph
  • Straight-line code: Always executes sequentially
  • Decision outcome: Depends on specific condition
Branch Coverage = (Branches Executed ÷ Total Branches) × 100%
Real-Life Example: Shopping Cart Discount Logic
Branch ID Type Path Test Required
B1-T Conditional quantity > 100 (TRUE) quantity = 150
B1-F Conditional quantity > 100 (FALSE) quantity = 50
B2-T Conditional isPremium (TRUE) isPremium = true
B2-F Conditional isPremium (FALSE) isPremium = false
U1 Unconditional return statement Any test case
Critical Exam Relationship:
• 100% Branch Coverage → 100% Statement Coverage ✅
• 100% Statement Coverage ↛ 100% Branch Coverage ❌

4.3.3 Value of White-Box Testing

Key Limitation: White-Box can test existing code with 100% coverage, but won't detect missing features like encryption, logging attempts, or account locking!
Comprehensive Testing Approach:
Black-Box + White-Box = Complete Testing
• Use White-Box for measuring coverage
• Use Black-Box for ensuring user requirements

4.4.1 Error Guessing

Real-Life Example: Web Application Security Testing (Fault Attacks)
Fault Category Potential Error Test Design Expected Result
Input Validation SQL Injection Username: admin'; DROP TABLE users;-- Input rejected safely
Session Management Session Hijacking Manipulate session tokens Unauthorized access prevented
Password Handling Weak Password Policy Password: "123" Weak password rejected
Error Messages Information Disclosure Wrong username vs wrong password Generic error messages

4.4.2 Exploratory Testing

Key Characteristic: Tests are designed, executed, and evaluated simultaneously while the tester learns about the test object.

Exploratory Testing Session Components

Component Description

Real-Life Example: When to Use Exploratory Testing
Situation Why Exploratory Testing? Example

4.5.1 Collaborative User Story Writing

User Story Definition: Represents a feature that will be valuable to the user or purchaser of the system.

The 3 C's of User Stories

C Component Description
Card Physical token Written description of the story
Conversation Dialogue Discussions about the story
Confirmation Agreement Acceptance criteria that confirm the story
Warning Sign: If stakeholders don't know how to test a user story, it may indicate the story is not ready for implementation.

4.5.2 Acceptance Criteria

Definition: Conditions that must be met for user story implementation to be accepted by stakeholders. Can be considered as test conditions that must be tested.
Usage Description Benefit Example
Define Scope تحديد النطاق Clarity in boundaries What should and shouldn't be implemented
Stakeholder Consensus إجماع أصحاب المصلحة Shared understanding Agreement on expectations
Scenarios السيناريوهات الإيجابية والسلبية Comprehensive coverage Normal and exceptional cases
Acceptance Testing أساس اختبار القبول Clear testing criteria Convert criteria to test cases
Planning التخطيط والتقدير Accurate estimates Calculate effort and time required

4.5.3 Acceptance Test-Driven Development (ATDD)

ATDD Definition: A test-first approach where test cases are created before implementing the user story by team members from different perspectives.
Key Concept: Examples ⟷ Tests
These terms are used interchangeably in ATDD

ATDD Process Integration

Test Technique Category Specific Techniques Application in ATDD Example
Black-box (4.2) EP, BVA, Decision Tables, State Transition Design test cases from acceptance criteria BVA for testing financial amount boundaries
White-box (4.3) Statement, Branch Testing Ensure required code coverage Branch coverage for business logic
Experience-based (4.4) Error Guessing, Exploratory, Checklist Add experience-based tests Error guessing for exceptional cases
Real-Life Example: Banking Transfer Feature
  • User Story: "As a bank customer, I want to transfer money..."
  • Executable Requirements: When test cases are captured in format supported by test automation framework
  • Collaboration: Customers + Developers + Testers work together

Final Exam Key Points:
• All techniques are complementary, not competitive
• 100% Branch Coverage always achieves 100% Statement Coverage
• Experience-based techniques fill gaps left by formal methods
• Collaboration-based approaches prevent defects rather than detect them
• ATDD creates executable requirements through examples
• The complete testing equation: Black-Box + White-Box + Experience-Based = Comprehensive Testing
