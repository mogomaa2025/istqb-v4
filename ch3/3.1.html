🌟 مقدمة: Static Testing Basics

"In contrast to dynamic testing, in static testing the software under test does not need to be executed. Code, process specification, system architecture specification or other work products are evaluated through manual examination (e.g., reviews) or with the help of a tool (e.g., static analysis). Test objectives include improving quality, detecting defects and assessing characteristics like readability, completeness, correctness, testability and consistency."
على عكس الاختبار الديناميكي، في الاختبار الساكن البرنامج تحت الاختبار لا يحتاج إلى تنفيذ. الكود، مواصفات العملية، مواصفات هندسة النظام أو منتجات العمل الأخرى يتم تقييمها من خلال الفحص اليدوي (مثل، المراجعات) أو بمساعدة أداة (مثل، التحليل الساكن). أهداف الاختبار تشمل تحسين الجودة، كشف العيوب وتقييم خصائص مثل القابلية للقراءة، الاكتمال، الصحة، القابلية للاختبار والاتساق.

Static وDynamic Testing يكملان بعضهما البعض ولهما أهداف مشابهة لكن مناهج مختلفة. كلاهما ضروري للحصول على تغطية شاملة وجودة عالية.

📄 3.1.1 Work Products Examinable by Static Testing

منتجات العمل القابلة للفحص بالاختبار الساكن

"Almost any work product can be examined using static testing. Examples include requirement specification documents, source code, test plans, test cases, product backlog items, test charters, project documentation, contracts and models. Any work product that can be read and understood can be the subject of a review. However, for static analysis, work products need a structure against which they can be checked."
تقريباً أي منتج عمل يمكن فحصه باستخدام الاختبار الساكن. الأمثلة تشمل وثائق مواصفات المتطلبات، الكود المصدري، خطط الاختبار، حالات الاختبار، عناصر product backlog، test charters، وثائق المشروع، العقود والنماذج. أي منتج عمل يمكن قراءته وفهمه يمكن أن يكون موضوع مراجعة. لكن، للـ التحليل الساكن، منتجات العمل تحتاج بنية يمكن فحصها ضدها.

💎 3.1.2 Value of Static Testing

قيمة وفوائد الاختبار الساكن

"Static testing can detect defects in the earliest phases of the SDLC, fulfilling the principle of early testing. It can also identify defects which cannot be detected by dynamic testing (e.g., unreachable code, design patterns not implemented as desired, defects in non-executable work products). Even though reviews can be costly to implement, the overall project costs are usually much lower than when no reviews are performed."
الاختبار الساكن يمكنه كشف العيوب في أقرب مراحل SDLC، محققاً مبدأ الاختبار المبكر. يمكنه أيضاً تحديد عيوب لا يمكن اكتشافها بالاختبار الديناميكي (مثل، الكود غير القابل للوصول، أنماط التصميم غير المنفذة كما هو مرغوب، عيوب في منتجات العمل غير القابلة للتنفيذ). رغم أن المراجعات قد تكون مكلفة للتطبيق، التكاليف الإجمالية للمشروع عادة أقل بكثير من عدم إجراء مراجعات.

⚔️ 3.1.3 Differences between Static and Dynamic Testing

الفروقات بين الاختبار الساكن والديناميكي

"Static testing and dynamic testing practices complement each other. They have similar objectives, such as supporting the detection of defects in work products, but there are also some differences: Static testing finds defects directly, while dynamic testing causes failures from which the associated defects are determined through subsequent analysis. Static testing can be applied to non-executable work products, while dynamic testing can only be applied to executable work products."
ممارسات الاختبار الساكن والديناميكي تكمل بعضها البعض. لديهما أهداف مشابهة، مثل دعم اكتشاف العيوب في منتجات العمل، لكن هناك أيضاً بعض الاختلافات: الاختبار الساكن يجد العيوب مباشرة، بينما الاختبار الديناميكي يسبب أعطال يتم تحديد العيوب المرتبطة بها من خلال تحليل لاحق. الاختبار الساكن يمكن تطبيقه على منتجات عمل غير قابلة للتنفيذ، بينما الاختبار الديناميكي يمكن تطبيقه فقط على منتجات عمل قابلة للتنفيذ.

🐛 Typical Defects Found by Static Testing

العيوب النموذجية التي يكتشفها الاختبار الساكن

"Typical defects that are easier and/or cheaper to find through static testing include: Defects in requirements (e.g., inconsistencies, ambiguities, contradictions, omissions, inaccuracies, duplications), Design defects (e.g., inefficient database structures, poor modularization), Certain types of coding defects (e.g., variables with undefined values, undeclared variables, unreachable or duplicated code, excessive code complexity), and Deviations from standards."
العيوب النموذجية التي أسهل و/أو أرخص للعثور عليها من خلال الاختبار الساكن تشمل: عيوب في المتطلبات (مثل، التضارب، الغموض، التناقضات، الحذف، عدم الدقة، التكرار)، عيوب التصميم (مثل، هياكل قاعدة البيانات غير الفعالة، سوء التعديل)، أنواع معينة من عيوب الترميز (مثل، متغيرات بقيم غير محددة، متغيرات غير مصرحة، كود غير قابل للوصول أو مكرر، تعقيد كود مفرط)، والانحرافات عن المعايير.

🎯 فئات العيوب التي يتفوق فيها Static Testing:

📋

Requirements Defects

عيوب في المتطلبات والمواصفات

  • Inconsistencies: تضارب بين المتطلبات
  • Ambiguities: غموض في الصياغة
  • Contradictions: تناقضات منطقية
  • Omissions: متطلبات مفقودة
  • Inaccuracies: معلومات غير دقيقة
  • Duplications: تكرار في المتطلبات
🎨

Design Defects

عيوب في التصميم والهندسة

  • Inefficient database structures: هياكل قاعدة بيانات غير فعالة
  • Poor modularization: تقسيم ضعيف للوحدات
  • Tight coupling: ربط قوي بين المكونات
  • Low cohesion: تماسك ضعيف داخلياً
  • Architectural inconsistencies: تضارب معماري
  • Design pattern violations: انتهاك أنماط التصميم
💻

Coding Defects

عيوب في الترميز والبرمجة

  • Undefined variables: متغيرات بقيم غير محددة
  • Undeclared variables: متغيرات غير مصرحة
  • Unreachable code: كود غير قابل للوصول
  • Duplicated code: كود مكرر
  • Excessive complexity: تعقيد مفرط
  • Dead code: كود ميت لا يستخدم
📏

Standards Violations

انتهاكات المعايير والاتفاقيات

  • Naming conventions: اتفاقيات التسمية
  • Coding standards: معايير البرمجة
  • Documentation standards: معايير التوثيق
  • Format standards: معايير التنسيق
  • Style guidelines: إرشادات الأسلوب
  • Comment conventions: اتفاقيات التعليقات
🔗

Interface Issues

مشاكل الواجهات والاتصال

  • Parameter mismatches: عدم تطابق المعاملات
  • Type inconsistencies: تضارب في الأنواع
  • Order problems: مشاكل في الترتيب
  • API misuse: سوء استخدام API
  • Protocol violations: انتهاك البروتوكولات
  • Contract breaches: خرق العقود البرمجية
🛡️

Security & Coverage

الأمان والتغطية

  • Buffer overflows: تجاوز سعة المخزن المؤقت
  • SQL injection vulnerabilities: ثغرات حقن SQL
  • Input validation gaps: ثغرات التحقق من المدخلات
  • Test coverage gaps: ثغرات تغطية الاختبار
  • Missing acceptance criteria tests: اختبارات معايير القبول المفقودة
  • Incomplete test scenarios: سيناريوهات اختبار ناقصة
