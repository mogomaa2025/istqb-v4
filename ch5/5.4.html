5.4 Configuration Management

إدارة التكوين - ISTQB Foundation Level v4.0

هدف التعلم: FL-5.4.1 (K2) Summarize how configuration management supports testing

ما هي إدارة التكوين (CM)؟

"In testing, configuration management (CM) provides a discipline for identifying, controlling, and tracking work products... as configuration items."

إدارة التكوين (CM) هي "نظام التحكم" الذي يضمن أننا نعرف دائمًا وبشكل دقيق ما الذي نختبره، وما هي الأدوات التي نستخدمها، وما هي إصدارات كل شيء.

تخيل أنك تختبر تطبيقًا. الأسئلة التي تجيب عليها CM هي:

  • ما هو إصدار التطبيق الذي أختبره الآن؟ (v1.2.1 أم v1.2.2؟)
  • ما هو إصدار نظام التشغيل؟ (Windows 11 Build X أم Build Y؟)
  • ما هي نسخة قاعدة البيانات المستخدمة؟
  • ما هو إصدار حالات الاختبار التي أنفذها؟

بدون CM، تصبح بيئة الاختبار فوضوية، وتصبح نتائج الاختبار غير موثوقة وغير قابلة للتكرار.

مكونات إدارة التكوين

تتضمن CM ثلاثة أنشطة رئيسية تعمل معًا لدعم الاختبار:

Identifying (تحديد)

تحديد وتسمية كل عنصر بشكل فريد.

→

Controlling (تحكم)

إدارة التغييرات على هذه العناصر.

→

Tracking (تتبع)

تسجيل حالة وهوية كل عنصر بمرور الوقت.

العناصر التي تتم إدارتها (Configuration Items): تشمل كل منتجات العمل (work products) مثل test plans, test strategies, test conditions, test cases, test scripts, test results, test logs, and test reports.

كيف تدعم إدارة التكوين الاختبار؟

"To properly support testing, CM ensures the following:"
التتبع والتتبع (Traceability) "All configuration items... are uniquely identified, version controlled, tracked for changes, and related to other configuration items so that traceability can be maintained throughout the test process."
تضمن أن كل شيء له هوية فريدة وإصدار، مما يسمح لنا بتتبع العلاقات (مثل: أي حالة اختبار تغطي أي متطلب في أي إصدار من التطبيق).
الوضوح (Unambiguity) "All identified documentation and software items are referenced unambiguously in test documentation."
تضمن عدم وجود أي غموض. عندما يذكر تقرير عيب "فشل في الإصدار 2.1"، نعرف بالضبط ما هو "الإصدار 2.1".

الخطوط الأساسية (Baselines) و DevOps

"If the configuration item is approved for testing, it becomes a baseline and can only be changed through a formal change control process. Configuration management keeps a record of changed configuration items when a new baseline is created. It is possible to revert to a previous baseline to reproduce previous test results."

الخط الأساسي (Baseline) هو "لقطة" متفق عليها ومعتمدة من مجموعة من العناصر في وقت معين. على سبيل المثال، "إصدار 1.0" هو خط أساسي. أي تغيير عليه يجب أن يمر بعملية تحكم رسمية، وهذا ينتج خطًا أساسيًا جديدًا (مثل "إصدار 1.1"). القدرة على العودة إلى خط أساسي قديم أمر بالغ الأهمية لإعادة إنتاج الأخطاء القديمة والتحقق منها.

في الممارسات الحديثة:

"Continuous integration, continuous delivery, continuous deployment and the associated testing are typically implemented as part of an automated DevOps pipeline..., in which automated CM is normally included."

دراسة حالة: استخدام Git و Jenkins لإدارة التكوين

تعتبر أنظمة التحكم في الإصدارات مثل Git وأدوات CI/CD مثل Jenkins هي العمود الفقري لإدارة التكوين الحديثة.

السيناريو: مختبر يكتشف عيبًا حرجًا في بيئة الإنتاج.

# 1. المطور يحدد المشكلة
git checkout -b fix/bug-123 v2.5.1
# يقوم بإنشاء فرع جديد من الإصدار v2.5.1 بالضبط (Baseline)
# ... يكتب الكود لإصلاح الخلل ...
git commit -m "Fix critical bug #123 in payment module"

# 2. Jenkins (أداة CI/CD) يتولى الأمر
# Jenkins يكتشف التغيير، يقوم بسحب الكود، ويبني إصدارًا جديدًا للاختبار (e.g., v2.5.2-beta)
# يقوم بتشغيل جميع الاختبارات الآلية على هذا الإصدار المحدد.

# 3. المختبر يقوم بالتحقق
# المختبر يتلقى إشعارًا بأن الإصدار v2.5.2-beta جاهز للاختبار.
# هو يعرف بالضبط ما هو الإصدار الذي يختبره (unambiguous)
# ويعرف بالضبط ما هو التغيير الذي يحتويه (traceability).

# إذا فشلت الاختبارات، يمكن دائمًا العودة إلى الإصدار v2.5.1 (revert to baseline).

هذا يوضح كيف تضمن أدوات CM الحديثة أن عملية الاختبار منظمة وموثوقة وقابلة للتكرار.
