🌟 مقدمة: Test Levels & Test Types

"Test levels are groups of test activities that are organized and managed together. Each test level is an instance of the test process, performed in relation to software at a given stage of development, from individual components to complete systems. Test types are groups of test activities related to specific quality characteristics and most of those test activities can be performed at every test level."
مستويات الاختبار هي مجموعات من أنشطة الاختبار التي يتم تنظيمها وإدارتها معاً. كل مستوى اختبار هو نموذج من عملية الاختبار، يتم تنفيذه بالنسبة للبرامج في مرحلة معينة من التطوير، من المكونات الفردية إلى الأنظمة الكاملة. أنواع الاختبار هي مجموعات من أنشطة الاختبار المتعلقة بخصائص جودة محددة ومعظم هذه الأنشطة يمكن تنفيذها في كل مستوى اختبار.

📊 2.2.1 Test Levels

مستويات الاختبار الخمسة

"In this syllabus, the following five test levels are described: Component testing (also known as unit testing), Component integration testing, System testing, System integration testing, and Acceptance testing. Test levels are distinguished by attributes: Test object, Test objectives, Test basis, Defects and failures, Approach and responsibilities."
في هذا المنهج، يتم وصف خمسة مستويات اختبار التالية: Component testing (يُعرف أيضاً بـ unit testing)، Component integration testing، System testing، System integration testing، وAcceptance testing. مستويات الاختبار تتميز بالصفات: كائن الاختبار، أهداف الاختبار، أساس الاختبار، العيوب والأعطال، النهج والمسؤوليات.

Component Testing

  • Individual components/modules

  • Database procedures

  • وظائف Component صحيحة
  • منطق الكود سليم
  • Code coverage مناسب
👥 المسؤولية:

  • Test harnesses أو frameworks
  • Automated unit tests

Component Integration

اختبار التفاعل بين المكونات المترابطة

🎯 Test Object:
  • Interfaces between components
  • Integrated modules
  • API calls and responses
📋 Test Objectives:
  • Interface صحيح بين Modules
  • Data flow سليم
  • Integration strategy فعالة
🔧 Integration Strategy:
  • Bottom-up approach
  • Top-down approach
  • Big-bang approach

اختبار النظام الكامل والسلوك الشامل

  • End-to-end functionality
  • System configurations
📋 Test Objectives:
  • End-to-end tasks تعمل
  • Non-functional requirements
  • System behavior شامل
👥 المسؤولية:
  • Independent test team
  • Representative test environment
  • System specifications

System Integration

اختبار التكامل مع الأنظمة والخدمات الخارجية

📋 Test Objectives:
  • External interfaces تعمل

🌍 البيئة:
  • Production-like environment

التحقق من جاهزية النظام والتحقق من احتياجات المستخدمين

  • System readiness للنشر
📝 الأنواع:

🎯 2.2.2 Test Types

أنواع الاختبار الأربعة

"A lot of test types exist and can be applied in projects. In this syllabus, the following four test types are addressed: Functional testing evaluates the functions that a component or system should perform. Non-functional testing evaluates attributes other than functional characteristics. Black-box testing is specification-based. White-box testing is structure-based."
توجد الكثير من أنواع الاختبار ويمكن تطبيقها في المشاريع. في هذا المنهج، يتم تناول أربعة أنواع اختبار التالية: Functional testing يقيم الوظائف التي يجب أن ينفذها المكون أو النظام. Non-functional testing يقيم الخصائص غير الوظيفية. Black-box testing مبني على المواصفات. White-box testing مبني على البنية.

🎭 الأنواع الأربعة للاختبار:

⚙️

Functional Testing

يقيم "ماذا" يجب أن يفعل النظام
اختبار الوظائف والميزات الأساسية
🎯 الأهداف الرئيسية:
  • • Functional Completeness: اكتمال الوظائف
  • • Functional Correctness: صحة الوظائف
  • • Functional Appropriateness: ملاءمة الوظائف
📋 أمثلة:
  • • اختبار تسجيل الدخول
  • • اختبار عمليات الشراء
  • • اختبار البحث والفلترة
  • • اختبار التقارير
📊

Non-Functional Testing

يقيم "كيف" يتصرف النظام
اختبار خصائص الجودة غير الوظيفية
🎯 الهدف الرئيسي:

فحص خصائص الجودة غير الوظيفية حسب معيار ISO/IEC 25010

📋 أمثلة:
  • • Performance Testing
  • • Security Testing
  • • Usability Testing
  • • Load Testing
⚫

Black-box Testing

Specification-based Testing
اختبار بناءً على المواصفات الخارجية
🎯 الهدف الرئيسي:

فحص سلوك النظام مقابل مواصفاته دون معرفة البنية الداخلية

📚 المصادر:
  • • Requirements documents
  • • User stories
  • • System specifications
  • • Use cases
⚪

White-box Testing

Structure-based Testing
اختبار بناءً على البنية الداخلية
🎯 الهدف الرئيسي:

تغطية البنية الأساسية بالاختبارات لمستوى مقبول

🧰 المصادر:
  • • Source code
  • • Architecture diagrams
  • • Work flows
  • • Data flows

🔄 تطبيق شامل

🎯 تركيز مختلف

🛠️ تقنيات متنوعة

تقنيات اختبار مختلفة لاشتقاق شروط وحالات الاختبار

معيار ISO/IEC 25010 يوفر تصنيفاً شاملاً لـ 7 خصائص أساسية للجودة غير الوظيفية

⚡ Performance Efficiency

قدرة النظام على الأداء الفعال مع استغلال الموارد المناسب

🔗 Compatibility

قدرة النظام على التوافق مع منتجات أو أنظمة أخرى

👥 Usability

قدرة النظام على الاستخدام بفعالية وكفاءة من قبل المستخدمين

🛡️ Reliability

قدرة النظام على أداء وظائفه تحت ظروف محددة لفترة معينة

🔒 Security

🔧 Maintainability

سهولة تعديل وتصحيح وتحسين النظام

📦 Portability

قدرة النظام على النقل من بيئة إلى أخرى

🔄 2.2.3 Confirmation & Regression Testing

اختبار التأكيد واختبار الانحدار

"Confirmation testing confirms that an original defect has been successfully fixed. Regression testing confirms that no adverse consequences have been caused by a change, including a fix that has already been confirmation tested. Regression test suites are run many times and generally the number of regression test cases will increase with each iteration or release, so regression testing is a strong candidate for automation."
اختبار التأكيد يؤكد أن العيب الأصلي تم إصلاحه بنجاح. اختبار الانحدار يؤكد أن لا توجد عواقب سلبية نتجت عن التغيير، بما في ذلك الإصلاح الذي تم اختبار تأكيده بالفعل. مجموعات اختبار الانحدار تُشغل مرات كثيرة وعموماً عدد حالات اختبار الانحدار سيزيد مع كل تكرار أو إطلاق، لذلك اختبار الانحدار مرشح قوي للأتمتة.

🔍 مقارنة شاملة: Confirmation vs Regression Testing

✅

Confirmation Testing

يؤكد إصلاح العيب الأصلي

🎯 الهدف:
  • التأكد من إصلاح Defect معين
  • إعادة تشغيل Test cases الفاشلة
  • التحقق من عمل الإصلاح
🛠️ الطرق:
  • تشغيل Test cases التي فشلت بسبب العيب
  • إضافة اختبارات جديدة لتغطية الإصلاحات
  • عندما الوقت/المال محدود: اختبار خطوات الإعادة فقط
⏰ التوقيت:
  • بعد إصلاح Developer للعيب
  • قبل Regression Testing
  • في نفس Test level للعيب
VS
🔄

Regression Testing

يؤكد عدم وجود عواقب سلبية

🎯 الهدف:
  • ضمان عدم تأثير التغييرات سلبياً
  • فحص نفس Component والمرتبطة
  • فحص الأنظمة المتصلة
🛠️ الطرق:
  • Impact Analysis لتحديد النطاق
  • تشغيل مجموعة اختبارات واسعة
  • فحص البيئة أيضاً، ليس فقط Test object
🤖 الأتمتة:
  • مرشح قوي للأتمتة
  • Test suites تزيد مع كل Release
  • CI/CD integration مهم
