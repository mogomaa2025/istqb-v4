🌟 مقدمة: SDLC وعلاقته بـ Testing

"A software development lifecycle (SDLC) model is an abstract, high-level representation of the software development process. A SDLC model defines how different development phases and types of activities performed within this process relate to each other, both logically and chronologically. Examples of SDLC models include: sequential development models (e.g., waterfall model, V-model), iterative development models (e.g., spiral model, prototyping), and incremental development models (e.g., Unified Process)."
نموذج دورة تطوير البرامج (SDLC) هو تمثيل مجرد عالي المستوى لعملية تطوير البرامج. نموذج SDLC يحدد كيف ترتبط مراحل التطوير المختلفة وأنواع الأنشطة المختلفة ببعضها البعض منطقياً وزمنياً. أمثلة على نماذج SDLC تشمل: النماذج التسلسلية (مثل Waterfall، V-model)، النماذج التكرارية (مثل Spiral، Prototyping)، والنماذج التدريجية (مثل Unified Process).

🔄 أنواع SDLC Models الرئيسية:

📏 Sequential Models
Waterfall, V-Model
🔄 Iterative Models
Spiral, Prototyping
📈 Incremental Models
Unified Process, RUP
🚀 Agile Models
Scrum, XP, Kanban

💥 2.1.1 Impact of SDLC on Testing

تأثير دورة تطوير البرامج على الاختبار

"Testing must be adapted to the SDLC to succeed. The choice of the SDLC impacts on the: Scope and timing of test activities, Level of detail of test documentation, Choice of test techniques and test approach, Extent of test automation, Role and responsibilities of a tester."
يجب تكييف Testing مع SDLC لتحقيق النجاح. اختيار SDLC يؤثر على: نطاق وتوقيت أنشطة الاختبار، مستوى تفاصيل وثائق الاختبار، اختيار تقنيات ونُهج الاختبار، مدى أتمتة الاختبار، دور ومسؤوليات المختبر.

✨ 2.1.2 Good Testing Practices

ممارسات Testing الجيدة العامة

"Good testing practices, independent of the chosen SDLC model, include the following: For every software development activity, there is a corresponding test activity, Different test levels have specific and different test objectives, Test analysis and design for a given test level begins during the corresponding development phase, Testers are involved in reviewing work products as soon as drafts are available."
ممارسات Testing الجيدة، بغض النظر عن نموذج SDLC المختار، تشمل التالي: لكل نشاط تطوير برامج، يوجد نشاط اختبار مقابل، مستويات الاختبار المختلفة لها أهداف محددة ومختلفة، تحليل وتصميم الاختبار يبدأ خلال مرحلة التطوير المقابلة، المختبرون يشاركون في مراجعة منتجات العمل بمجرد توفر المسودات.

🌍 أفضل الممارسات العامة - مطبقة في جميع SDLC:

🔄 Corresponding Activities

لكل نشاط تطوير، يوجد نشاط testing مقابل - Quality Control شامل

🎯 Specific Objectives

مستويات Testing مختلفة لها أهداف محددة - تجنب التكرار

⏰ Early Test Design

Test Analysis والـ Design يبدآن مع مرحلة التطوير المقابلة

👥 Early Involvement

Testers يشاركون في مراجعة المستندات فور توفرها

📋 شرح تفصيلي للممارسات:

🔄 المبدأ الأول:

Development Activity → Test Activity
• Requirements → Test Analysis
• Design → Test Design
• Coding → Test Implementation
• Deployment → Test Execution

🎯 المبدأ الثاني:

Test Levels لها أهداف مختلفة
• Unit Testing → Code Logic
• Integration → Interface Issues
• System → End-to-end
• Acceptance → User Needs

🥇 2.1.3 Test-First Approaches

نُهج Test-First في التطوير

"TDD, ATDD and BDD are similar development approaches, where tests are defined as a means of directing development. Each of these approaches implements the principle of early testing and follows a shift-left approach, since the tests are defined before the code is written. They support an iterative development model."
TDD، ATDD، وBDD هي نُهج تطوير متشابهة، حيث تُعرَّف الاختبارات كوسيلة لتوجيه التطوير. كل من هذه النُهج تطبق مبدأ الاختبار المبكر وتتبع نهج shift-left، حيث تُعرَّف الاختبارات قبل كتابة الكود. تدعم هذه النُهج نموذج التطوير التكراري.

🔄 مقارنة شاملة: TDD vs ATDD vs BDD

⚗️

Test-Driven Development (TDD)

🎯 التركيز:

توجيه الـ Coding من خلال Test Cases بدلاً من التصميم المكثف

🔄 العملية:
  • كتابة Test أولاً
  • كتابة الـ Code لتمرير الـ Test
  • Refactoring للكود والـ Tests
  • تكرار الدورة: Red-Green-Refactor
👥 المسؤولية:

المطورون - تركيز على Unit Testing والكود

✅

Acceptance Test-Driven Development (ATDD)

🎯 التركيز:

اشتقاق Tests من Acceptance Criteria كجزء من عملية تصميم النظام

🔄 العملية:
  • تحديد Acceptance Criteria
  • كتابة Acceptance Tests
  • تطوير التطبيق لتمرير Tests
  • التحقق من تلبية المتطلبات
👥 المسؤولية:

Business Analysts + Testers - تركيز على User Requirements

🎭

Behavior-Driven Development (BDD)

🎯 التركيز:

التعبير عن السلوك المطلوب بـ Test Cases بلغة طبيعية مفهومة للجميع

🔄 العملية:
  • كتابة Scenarios بـ Given/When/Then
  • ترجمة تلقائية إلى Executable Tests
  • تطوير الكود لتمرير السيناريوهات
  • تحقيق السلوك المطلوب
👥 المسؤولية:

All Stakeholders - لغة مفهومة للجميع

📝 مثال BDD Scenario:

Feature: User Login

Scenario: Successful login with valid credentials
  Given the user is on the login page
  When the user enters valid username and password
  Then the user should be redirected to the dashboard
  And a welcome message should be displayed

هذا السيناريو مفهوم لجميع أصحاب المصلحة ويمكن ترجمته تلقائياً إلى اختبارات قابلة للتنفيذ

🌟 الفوائد المشتركة للثلاث نُهج:

⏰ Early Testing
اختبار مبكر في دورة التطوير
🔄 Shift-Left
نقل Testing إلى بداية العملية
🔁 Iterative
دعم نماذج التطوير التكرارية
🤖 Automation
اختبارات آلية لضمان الجودة

🚀 2.1.4 DevOps and Testing

DevOps وتأثيره على Testing

"DevOps is an organizational approach aiming to create synergy by getting development (including testing) and operations to work together to achieve a set of common goals. DevOps requires a cultural shift within an organization to bridge the gaps between development and operations while treating their functions with equal value. DevOps promotes team autonomy, fast feedback, integrated toolchains, and technical practices like continuous integration (CI) and continuous delivery (CD)."
DevOps هو نهج تنظيمي يهدف إلى خلق تآزر من خلال جعل التطوير (بما يشمل الاختبار) والعمليات يعملان معاً لتحقيق مجموعة من الأهداف المشتركة. DevOps يتطلب تحولاً ثقافياً داخل المنظمة لسد الفجوات بين التطوير والعمليات مع معاملة وظائفهما بقيمة متساوية. DevOps يعزز استقلالية الفريق، التغذية الراجعة السريعة، سلاسل الأدوات المتكاملة، والممارسات التقنية مثل CI وCD.

✅ فوائد DevOps للـ Testing

  • Fast Feedback: تغذية راجعة سريعة على جودة الكود والتغييرات
  • Shift-Left Approach: CI يعزز نهج shift-left في الاختبار
  • Automated Processes: أتمتة CI/CD تسهل بيئات اختبار مستقرة
  • Non-Functional Focus: زيادة التركيز على خصائص الجودة غير الوظيفية
  • Reduced Manual Testing: الأتمتة تقلل الحاجة للاختبار اليدوي المتكرر
  • Regression Risk: تقليل مخاطر Regression بالاختبارات الآلية الواسعة
VS

⚠️ تحديات DevOps

  • Pipeline Definition: يجب تعريف وإنشاء DevOps delivery pipeline
  • Tool Management: أدوات CI/CD تحتاج إدخال وصيانة
  • Automation Resources: أتمتة الاختبار تحتاج موارد إضافية
  • Maintenance Complexity: صعوبة إنشاء وصيانة الأتمتة
  • Initial Investment: استثمار كبير في البداية للأدوات والتدريب
  • Cultural Shift: تحول ثقافي مطلوب في المنظمة

🔧 تكامل Testing في CI/CD:

🧪 Unit Tests

تشغل مع كل commit - فحص سريع للوظائف الأساسية

🔗 Integration Tests

في مرحلة Build - التحقق من تكامل المكونات

🌐 System Tests

في بيئة Staging - اختبار النظام الكامل

📊 Performance Tests

في Pipeline - مراقبة الأداء والاستقرار

⚠️ ملاحظة مهمة:

رغم المستوى العالي من الأتمتة في DevOps، Manual Testing - خاصة من منظور المستخدم - سيظل مطلوباً

⬅️ 2.1.5 Shift-Left Approach

نهج Shift-Left في الاختبار

"The principle of early testing is sometimes referred to as shift-left because it is an approach where testing is performed earlier in the SDLC. Shift-left normally suggests that testing should be done earlier (e.g., not waiting for code to be implemented or for components to be integrated), but it does not mean that testing later in the SDLC should be neglected."
مبدأ الاختبار المبكر يُشار إليه أحياناً بـ shift-left لأنه نهج حيث يتم تنفيذ الاختبار مبكراً في SDLC. Shift-left عادة يقترح أن الاختبار يجب أن يتم مبكراً (مثلاً، عدم انتظار تنفيذ الكود أو تكامل المكونات)، لكنه لا يعني أن الاختبار المتأخر في SDLC يجب إهماله.

📊 تطور Shift-Left عبر SDLC:

📋 Requirements Phase

Traditional: مراجعة المتطلبات
Shift-Left: مراجعة المواصفات من منظور Testing + البحث عن الغموض والتناقضات

🎨 Design Phase

Traditional: مراجعة التصميم
Shift-Left: كتابة Test Cases قبل الكود + تشغيل في Test Harness أثناء التنفيذ

💻 Development Phase

Traditional: كتابة الكود
Shift-Left: استخدام CI/CD + تغذية راجعة سريعة + Component Tests مع كل commit

🔍 Before Dynamic Testing

Traditional: بداية Testing
Shift-Left: Static Analysis للكود + Non-functional Testing على مستوى Component

🧪 Testing Phase

Traditional: اختبار شامل متأخر
Shift-Left: اختبار مستمر + عمليات آلية + بيئات اختبار مستقرة

🛠️ ممارسات Shift-Left الأساسية:

📋 Specification Review

مراجعة المواصفات من منظور Testing للعثور على الغموض والنقص والتناقضات

✍️ Test-First Writing

كتابة Test Cases قبل الكود وتشغيلها في Test Harness أثناء التنفيذ

🔗 CI/CD Integration

استخدام CI وCD للحصول على تغذية راجعة سريعة مع Component Tests آلية

🔍 Static Analysis

إجراء Static Analysis للكود قبل Dynamic Testing أو كجزء من عملية آلية

⚡ Early Non-Functional

تنفيذ Non-functional Testing من مستوى Component بدلاً من انتظار النظام الكامل

💰 تحليل التكلفة والعائد:

📈 تكاليف إضافية مبكراً
  • • تدريب الفريق
  • • جهد إضافي في البداية
  • • استثمار في الأدوات
  • • تغيير في العمليات
→
💎 توفيرات لاحقاً
  • • تقليل تكلفة إصلاح العيوب
  • • توفير جهد ووقت
  • • جودة أعلى للمنتج النهائي
  • • رضا أكبر للعملاء

🤝 أهمية إقناع أصحاب المصلحة:

للنجاح في نهج Shift-Left، من المهم أن يكون أصحاب المصلحة مقتنعين ومؤيدين لهذا المفهوم
يحتاج تغيير في الثقافة والعقلية والاستثمار في التدريب والأدوات

🔄 2.1.6 Retrospectives and Process Improvement

Retrospectives وتحسين العمليات

"Retrospectives (also known as 'post-project meetings' and project retrospectives) are often held at the end of a project or an iteration, at a release milestone, or can be held when needed. In these meetings the participants discuss: What was successful, and should be retained? What was not successful and could be improved? How to incorporate the improvements and retain the successes in the future?"
Retrospectives (تُعرف أيضاً بـ 'اجتماعات ما بعد المشروع' ومراجعات المشروع) تُعقد غالباً في نهاية مشروع أو تكرار، في معلم إطلاق، أو يمكن عقدها عند الحاجة. في هذه الاجتماعات يناقش المشاركون: ما كان ناجحاً، ويجب الاحتفاظ به؟ ما لم يكن ناجحاً ويمكن تحسينه؟ كيف ندمج التحسينات ونحتفظ بالنجاحات في المستقبل؟

🔄 عملية Retrospectives:

❓ الأسئلة الثلاثة الأساسية:

✅ What was successful?

ما الذي كان ناجحاً ويجب الاحتفاظ به؟

❌ What needs improvement?

ما الذي لم يكن ناجحاً ويمكن تحسينه؟

🚀 How to improve?

كيف ندمج التحسينات ونحتفظ بالنجاحات؟

👥 المشاركون في Retrospectives:

🧪 Testers
💻 Developers
🏗️ Architects
👔 Product Owner
📊 Business Analysts

⏰ توقيت وتوثيق Retrospectives:

📅 التوقيت:
  • • نهاية المشروع
  • • نهاية التكرار (Iteration)
  • • معالم الإطلاق (Release Milestones)
  • • عند الحاجة (Ad-hoc)
📝 التوثيق:
  • • تسجيل النتائج
  • • جزء من Test Completion Report
  • • خطط التحسين
  • • متابعة التطبيق
