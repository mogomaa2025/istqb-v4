2.1.3 Advanced Development Approaches

Test-Driven Development (TDD)

TDD Philosophy: توجيه الـ Coding من خلال Test Cases بدلاً من التصميم المكثف (Guide coding through test cases instead of intensive design)
TDD Phase Activity Focus Outcome
Red Write failing test Define expected behavior Test that fails initially
Green Write minimal code to pass Make test pass with simplest solution Working code that passes test
Refactor Improve code quality Clean up while maintaining functionality Clean, maintainable code
TDD Key Players: المطورون - تركيز على Unit Testing والكود (Developers - focus on Unit Testing and code)

Acceptance Test-Driven Development (ATDD)

ATDD Philosophy: اشتقاق Tests من Acceptance Criteria كجزء من عملية تصميم النظام (Derive tests from acceptance criteria as part of system design process)
ATDD Key Players: Business Analysts + Testers - تركيز على User Requirements (focus on User Requirements)

Behavior-Driven Development (BDD)

BDD Philosophy: التعبير عن السلوك المطلوب بـ Test Cases بلغة طبيعية مفهومة للجميع (Express required behavior with test cases in natural language understandable by everyone)
BDD Key Players: All Stakeholders - لغة مفهومة للجميع (Language understandable by everyone)
Real-Life BDD Example: E-commerce Login Feature
Gherkin Scenario:
Given I am on the login page
When I enter valid username and password
And I click the login button
Then I should be redirected to the dashboard
And I should see a welcome message
This scenario is understandable by all stakeholders and can be automatically translated into executable tests.

2.1.4 DevOps and Testing

DevOps Testing Approach: High automation with continuous testing throughout the CI/CD pipeline, but manual testing from user perspective remains essential.
Pipeline Stage Testing Activity Purpose Automation Level
Commit Stage تشغل مع كل commit - فحص سريع للوظائف الأساسية Fast check of basic functionality 100% Automated
Build Stage في مرحلة Build - التحقق من تكامل المكونات Component integration verification 100% Automated
Staging في بيئة Staging - اختبار النظام الكامل Complete system testing Mostly Automated
Production Pipeline في Pipeline - مراقبة الأداء والاستقرار Performance and stability monitoring Automated + Manual
Important DevOps Principle: رغم المستوى العالي من الأتمتة في DevOps، Manual Testing - خاصة من منظور المستخدم - سيظل مطلوباً (Despite high automation in DevOps, manual testing - especially from user perspective - will remain required)

2.1.5 Shift-Left Testing Approach

Shift-Left Concept: Moving testing activities earlier in the development lifecycle to detect and prevent defects sooner, reducing cost and improving quality.

Traditional vs Shift-Left Testing Comparison

Activity Traditional Approach Shift-Left Approach Benefit
Requirements Review مراجعة المتطلبات مراجعة المواصفات من منظور Testing + البحث عن الغموض والتناقضات Early requirement defect detection
Design Review مراجعة التصميم كتابة Test Cases قبل الكود + تشغيل في Test Harness أثناء التنفيذ Test-driven design validation
Coding Phase كتابة الكود استخدام CI/CD + تغذية راجعة سريعة + Component Tests مع كل commit Continuous quality feedback
Testing Start بداية Testing Static Analysis للكود + Non-functional Testing على مستوى Component Early quality assessment
System Testing اختبار شامل متأخر اختبار مستمر + عمليات آلية + بيئات اختبار مستقرة Continuous validation

Five Key Shift-Left Practices

Practice Description Implementation Impact
Early Specification Review مراجعة المواصفات من منظور Testing للعثور على الغموض والنقص والتناقضات Review specifications from testing perspective to find ambiguity and contradictions Prevent requirement defects
Test-First Development كتابة Test Cases قبل الكود وتشغيلها في Test Harness أثناء التنفيذ Write test cases before code and run in test harness during implementation Code driven by tests
CI/CD Integration استخدام CI وCD للحصول على تغذية راجعة سريعة مع Component Tests آلية Use CI and CD for fast feedback with automated component tests Rapid quality feedback
Static Analysis إجراء Static Analysis للكود قبل Dynamic Testing أو كجزء من عملية آلية Perform static analysis before dynamic testing or as part of automated process Early code quality assessment
Early Non-functional Testing تنفيذ Non-functional Testing من مستوى Component بدلاً من انتظار النظام الكامل Execute non-functional testing at component level instead of waiting for complete system Early performance insights
Success Requirements: للنجاح في نهج Shift-Left، من المهم أن يكون أصحاب المصلحة مقتنعين ومؤيدين لهذا المفهوم (For Shift-Left success, stakeholders must be convinced and supportive of this concept). يحتاج تغيير في الثقافة والعقلية والاستثمار في التدريب والأدوات (Requires culture and mindset change and investment in training and tools).
Real-Life Example: Microsoft's Shift-Left Implementation
Microsoft implemented shift-left testing across their development teams:
  • Early Review: Security requirements reviewed by security testers during design phase
  • Test-First: Azure services use TDD approach with automated unit tests written before implementation
  • CI/CD: Every commit triggers automated test suite within 10 minutes
  • Static Analysis: Code quality gates prevent poor code from reaching main branch
  • Early Performance: Performance tests run on individual microservices, not just complete systems
Result: 70% reduction in production defects and 40% faster release cycles.

2.2.1 Test Levels Overview

Test Levels Framework: WHEN & WHERE (متى وأين نختبر؟) vs WHAT & HOW (ماذا وكيف نختبر؟) - Development phases vs Quality characteristics

Five Main Test Levels

Test Level Arabic Description Primary Focus Typical Test Objects Key Defects Found
Component Testing اختبار المكونات الفردية بمعزل عن بقية النظام Individual components in isolation Units, modules, classes Logic errors, data handling issues
Integration Testing اختبار التفاعل بين المكونات المترابطة Interactions between integrated components Interfaces, subsystems Interface mismatches, data flow problems
System Testing اختبار النظام الكامل والسلوك الشامل Complete system and overall behavior Complete applications, systems End-to-end workflow issues
System Integration Testing اختبار التكامل مع الأنظمة والخدمات الخارجية Integration with external systems and services System of systems, external interfaces External interface failures, data exchange issues
Acceptance Testing التحقق من جاهزية النظام والتحقق من احتياجات المستخدمين System readiness and user needs verification Complete business processes, user workflows Usability issues, business rule violations

2.2.2 Test Types

Four Main Test Types: جميع الأنواع الأربعة يمكن تطبيقها في جميع مستويات الاختبار. التركيز يختلف في كل مستوى حسب الهدف والسياق (All four types can be applied at all test levels. Focus differs at each level according to objective and context)

Test Type Categories

Test Type Purpose Focus Area Techniques Used
Functional Testing فحص سلوك النظام مقابل مواصفاته دون معرفة البنية الداخلية What the system does Black-box techniques, specification-based
Non-functional Testing فحص خصائص الجودة غير الوظيفية حسب معيار ISO/IEC 25010 How well the system performs Specialized tools and techniques
White-box Testing تغطية البنية الأساسية بالاختبارات لمستوى مقبول Internal structure and implementation Structure-based techniques, coverage analysis
Change-related Testing Verification of changes and prevention of unintended consequences Change impact and regression Re-testing, regression testing

2.2.3 Non-functional Testing and ISO/IEC 25010

Seven Quality Characteristics (ISO/IEC 25010)

Quality Characteristic Arabic Description Definition Test Examples
Performance Efficiency قدرة النظام على الأداء الفعال مع استغلال الموارد المناسب System's ability to perform effectively with appropriate resource utilization Load testing, stress testing, volume testing
Compatibility قدرة النظام على التوافق مع منتجات أو أنظمة أخرى System's ability to be compatible with other products or systems Interoperability testing, co-existence testing
Usability قدرة النظام على الاستخدام بفعالية وكفاءة من قبل المستخدمين System's ability to be used effectively and efficiently by users User experience testing, accessibility testing
Reliability قدرة النظام على أداء وظائفه تحت ظروف محددة لفترة معينة System's ability to perform functions under specified conditions for a specified time Stability testing, recovery testing
Security قدرة النظام على حماية المعلومات والبيانات من الوصول غير المصرح System's ability to protect information and data from unauthorized access Vulnerability testing, penetration testing
Maintainability سهولة تعديل وتصحيح وتحسين النظام Ease of modifying, correcting, and improving the system Code analysis, changeability testing
Portability قدرة النظام على النقل من بيئة إلى أخرى System's ability to be transferred from one environment to another Installation testing, migration testing

Non-functional Testing Considerations

Consideration Description Implementation Example
Early Start أحياناً من المناسب البدء مبكراً في دورة الحياة Start early in lifecycle (reviews, component testing) Performance requirements review during design
Derived from Functional العديد من Non-functional tests مشتقة من functional tests Many non-functional tests derived from functional tests Load testing based on functional user scenarios
Late Discovery Risk الاكتشاف المتأخر للعيوب غير الوظيفية تهديد جدي لنجاح المشروع Late discovery of non-functional defects is serious threat Performance issues found only in production
Specialized Environment أحياناً يحتاج بيئة اختبار خاصة Sometimes requires special test environment Usability lab for user experience testing
Real-Life Example: Netflix Performance Testing Strategy
Netflix conducts comprehensive non-functional testing across all quality characteristics:
  • Performance: Simulates millions of concurrent users streaming video
  • Compatibility: Tests across 1000+ device types and configurations
  • Usability: A/B tests user interface changes with real users
  • Reliability: Chaos engineering to test system resilience
  • Security: Continuous security scanning and penetration testing
  • Maintainability: Code quality metrics and technical debt monitoring
  • Portability: Tests deployment across multiple cloud providers
This comprehensive approach ensures Netflix delivers high-quality streaming experience globally.

2.2.4 Confirmation and Regression Testing

Two-Phase Change Verification:
  • يؤكد إصلاح العيب الأصلي (Confirms original defect fix)
  • يؤكد عدم وجود عواقب سلبية (Confirms no negative consequences)

Confirmation Testing (Re-testing)

Aspect Description Purpose Scope
Objective Verify that original defect has been fixed Confirm successful defect resolution Specific defect area only
Test Cases Execute same test cases that originally failed Prove defect no longer exists Exact reproduction steps
Timing Immediately after defect fix Quick verification of fix Before broader testing

Regression Testing

Impact Analysis Recommendation: "من المستحسن أولاً إجراء تحليل التأثير لتحسين نطاق اختبار الانحدار. تحليل التأثير يُظهر أي أجزاء من البرنامج يمكن أن تتأثر" (It's advisable to first conduct impact analysis to improve regression test scope. Impact analysis shows which parts of software can be affected)

Impact Analysis Questions

Question Arabic Translation Analysis Focus Decision Impact
Which parts will be affected? أي أجزاء ستتأثر بالتغيير؟ Change scope analysis Test selection
What is the risk level? ما مستوى المخاطر لكل جزء؟ Risk assessment Prioritization
Which tests should be re-run? أي اختبارات يجب إعادة تشغيلها؟ Test case selection Execution planning

Regression Testing Benefits

Benefit Description Value
Prioritization تشغيل الأهم أولاً وتوفير الوقت Run most important first, save time
Speed تشغيل مئات الاختبارات في دقائق بدلاً من أيام Run hundreds of tests in minutes instead of days
Accuracy تجنب الأخطاء البشرية في الاختبارات المتكررة Avoid human errors in repetitive tests
Frequency إمكانية التشغيل المستمر مع كل build جديد Continuous execution with every new build
Coverage إمكانية اختبار سيناريوهات أكثر وأعمق Test more and deeper scenarios
CI/CD Integration: "عند استخدام CI (كما في DevOps)، من الممارسات الجيدة تضمين automated regression tests. حسب الحالة، قد يشمل هذا regression tests في مستويات مختلفة" (When using CI (as in DevOps), it's good practice to include automated regression tests. Depending on case, this may include regression tests at different levels)
Real-Life Example: Google's Regression Testing Strategy
Google maintains one of the largest automated regression test suites in the industry:
  • Scale: Over 100 million test cases executed daily
  • Speed: Results available within 10 minutes of code commit
  • Intelligence: Machine learning selects most relevant tests based on code changes
  • Coverage: Tests run at component, integration, and system levels
  • Impact Analysis: Automated analysis determines which tests to run for each change
This approach allows Google to maintain quality while deploying thousands of changes daily.

2.3.1 Maintenance Testing Context

Maintenance Testing Lifecycle Position

Key Timing: Maintenance Testing يحدث في مرحلة Production وما بعدها، عندما يكون النظام حي ويعمل مع المستخدمين الحقيقيين (Maintenance Testing occurs in Production phase and beyond, when system is live and working with real users)

Three Types of Maintenance (ISO 14764)

Maintenance Type Arabic Description Purpose Examples Frequency
Corrective إصلاح العيوب الموجودة Fix existing defects Bug fixes, security patches As needed when defects found
Adaptive تكيف مع البيئة الجديدة Adapt to new environment OS upgrades, new browsers Periodic with environment changes
Perfective تحسين الأداء والوظائف Improve performance and functions Performance optimization, new features Continuous throughout system life

Maintenance Testing Triggers

Trigger Category Description Examples Testing Approach
Modifications تعديلات وتحسينات النظام • New features
• Bug fixes
• Performance improvements		 Focused testing + selective regression
Upgrades ترقية البيئة التشغيلية • Operating system updates
• Database upgrades
• Framework migrations		 Compatibility testing + full regression
Retirement نهاية دورة حياة النظام • Data migration
• System decommissioning
• Archive processes		 Data integrity + migration testing

2.3.2 Risk Assessment for Maintenance

Three Key Risk Factors:
  • درجة خطر التغيير (Risk level of change)
  • حجم النظام الموجود (Size of existing system)
  • حجم التغيير (Size of change)
Risk Level System Size Change Size Testing Scope Example Scenario
High Large Major Extensive + Full Regression Core banking system architecture change
Medium Medium Minor Focused + Selective Regression E-commerce checkout process update
Low Small Patch Minimal + Smoke Testing Small business website bug fix

2.3.3 Impact Analysis in Maintenance Testing

Six Key Impact Analysis Questions:
Question Arabic Translation Analysis Purpose Decision Support
What parts will be affected? ما هي الأجزاء التي ستتأثر بالتغيير المقترح؟ Identify scope of change impact Test planning and resource allocation
Which components are connected? أي مكونات أخرى ترتبط بالجزء المتغير؟ Map dependencies and relationships Integration testing requirements
What is the risk level? ما مستوى المخاطر والعواقب المحتملة؟ Assess potential consequences Risk mitigation strategies
What is the cost-benefit? ما تكلفة التغيير مقابل الفائدة المتوقعة؟ Economic impact evaluation Go/no-go decision making
What tests are required? أي اختبارات مطلوبة للتحقق من سلامة التغيير؟ Define testing requirements Test strategy and execution planning
Should we proceed? هل يجب المضي قدماً بالتغيير أم لا؟ Final decision recommendation Change approval process
