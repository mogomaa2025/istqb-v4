🧪 1.1 What is Testing?

إيه هو Testing؟ - الشرح الكامل والشامل

🌟 مقدمة: إيه هو Software Testing؟

"Software systems are an integral part of our daily life. Most people have had experience with software that did not work as expected. Software testing assesses software quality and helps reducing the risk of software failure in operation. Software testing is a set of activities to discover defects and evaluate the quality of software artifacts."
الأنظمة البرمجية جزء لا يتجزأ من حياتنا اليومية. معظم الناس مروا بتجربة برامج لم تعمل كما هو متوقع. Software Testing بيقيم جودة البرامج ويساعد في تقليل مخاطر فشل البرامج أثناء التشغيل. Software Testing هو مجموعة من الأنشطة لاكتشاف العيوب وتقييم جودة منتجات البرامج.
🧪 Software Testing = مجموعة شاملة من الأنشطة لاكتشاف العيوب وتقييم جودة البرامج
🎯 الهدف الأساسي

تقييم جودة البرامج وتقليل مخاطر الفشل في التشغيل

🔍 اكتشاف العيوب

العثور على الأخطاء والمشاكل قبل وصولها للمستخدمين

📊 تقييم الجودة

قياس مدى مطابقة البرنامج للمتطلبات والمعايير

🛡️ إدارة المخاطر

تقليل احتمالية حدوث مشاكل في البيئة التشغيلية

❌ المفاهيم الخاطئة الشائعة

"A common misconception about testing is that it only consists of executing tests (i.e., running the software and checking the test results). However, software testing also includes other activities and must be aligned with the software development lifecycle. Another common misconception about testing is that testing focuses entirely on verifying the test object."
من المفاهيم الخاطئة الشائعة حول Testing إنه بيتكون فقط من تشغيل الاختبارات (يعني تشغيل البرنامج وفحص النتائج). لكن Software Testing كمان بيشمل أنشطة تانية ولازم يكون متماشي مع دورة تطوير البرامج. مفهوم خاطئ تاني إن Testing بيركز بالكامل على التحقق من الكائن المختبر.

🚫 تصحيح المفاهيم الخاطئة:

🤖
المفهوم الخاطئ الأول
❌ "Testing = تشغيل البرنامج فقط"
✅ Testing شامل: التخطيط، التصميم، التنفيذ، المراقبة، التقييم، والتوثيق
🔍
المفهوم الخاطئ الثاني
❌ "Testing = Verification فقط"
✅ Testing = Verification (هل بنينا المنتج صح؟) + Validation (هل بنينا المنتج الصح؟)
⚡
المفهوم الخاطئ الثالث
❌ "Testing نشاط تقني فقط"
✅ Testing يحتاج تخطيط، إدارة، تقدير، مراقبة، تحكم + مهارات تحليلية وتفكير نقدي

⚖️ أنواع Testing: Static vs Dynamic

"Testing may be dynamic or static. Dynamic testing involves the execution of software, while static testing does not. Static testing includes reviews and static analysis. Dynamic testing uses different types of test techniques and test approaches to derive test cases."
Testing ممكن يكون ديناميكي أو ثابت. Dynamic Testing بيشمل تشغيل البرنامج، بينما Static Testing لا. Static Testing بيشمل المراجعات والتحليل الثابت. Dynamic Testing بيستخدم أنواع مختلفة من تقنيات ونُهج الاختبار لاشتقاق حالات الاختبار.

🔄 مقارنة شاملة: Static vs Dynamic Testing

📋

Static Testing

اختبار بدون تشغيل البرنامج - فحص الكود والوثائق والتصميم
🛠️ أمثلة:
  • • Code Reviews (مراجعة الكود)
  • • Requirements Reviews (مراجعة المتطلبات)
  • • Design Reviews (مراجعة التصميم)
  • • Static Analysis Tools (أدوات التحليل الثابت)
  • • Walkthroughs (جولات فحص)
  • • Inspections (عمليات تفتيش)
VS
▶️

Dynamic Testing

اختبار مع تشغيل البرنامج - تنفيذ الكود ومراقبة السلوك
🎮 أمثلة:
  • • Functional Testing (اختبار وظيفي)
  • • Performance Testing (اختبار الأداء)
  • • Security Testing (اختبار الأمان)
  • • Usability Testing (اختبار سهولة الاستخدام)
  • • Integration Testing (اختبار التكامل)
  • • System Testing (اختبار النظام)

📊 مقارنة تفصيلية:

⏰ التوقيت

Static: مبكر في التطوير
Dynamic: بعد كتابة الكود

💰 التكلفة

Static: أقل تكلفة
Dynamic: أعلى تكلفة

🎯 الهدف

Static: منع العيوب
Dynamic: اكتشاف وإصلاح العيوب

📈 التغطية

Static: 100% تغطية البيانات
Dynamic: أقل من 50% تغطية

🎯 1.1.1 Test Objectives

الأهداف النموذجية للاختبار

"The typical test objectives are: Evaluating work products, Triggering failures and finding defects, Ensuring required coverage, Reducing the level of risk, Verifying requirements fulfillment, Verifying compliance, Providing information to stakeholders, Building confidence, and Validating completeness."
الأهداف النموذجية للاختبار هي: تقييم منتجات العمل، إثارة الأعطال واكتشاف العيوب، ضمان التغطية المطلوبة، تقليل مستوى المخاطر، التحقق من تحقيق المتطلبات، التحقق من الامتثال، توفير المعلومات لأصحاب المصلحة، بناء الثقة، والتحقق من الاكتمال.
📋

Evaluating Work Products

تقييم منتجات العمل مثل المتطلبات، قصص المستخدمين، التصاميم، والكود
🔍 أمثلة عملية:
  • • فحص وثائق المتطلبات للوضوح والاكتمال
  • • مراجعة User Stories للتأكد من قابليتها للاختبار
  • • تقييم تصاميم النظام للتناسق والجودة
  • • مراجعة الكود للتأكد من اتباع المعايير
🐛

Triggering Failures & Finding Defects

إثارة الأعطال واكتشاف العيوب في النظام قبل وصولها للمستخدمين
🎯 أمثلة عملية:
  • • اختبار الحالات الحدية التي تسبب تعطل النظام
  • • البحث عن أخطاء منطقية في الكود
  • • اكتشاف مشاكل الأداء تحت الضغط
  • • العثور على ثغرات أمنية محتملة
📊

Ensuring Required Coverage

ضمان التغطية المطلوبة لكائن الاختبار
📈 أمثلة عملية:
  • • تغطية 100% من المتطلبات الحرجة
  • • اختبار جميع مسارات الكود الأساسية
  • • فحص كل واجهات المستخدم
  • • اختبار جميع تكاملات النظام
⚠️

Reducing Risk Level

تقليل مستوى مخاطر جودة البرامج غير الكافية
🛡️ أمثلة عملية:
  • • تقليل مخاطر فقدان البيانات
  • • منع مشاكل الأمان والخصوصية
  • • تجنب فشل النظام في بيئة الإنتاج
  • • تقليل مخاطر عدم رضا المستخدمين
✅

Verifying Requirements

التحقق من أن المتطلبات المحددة قد تم تحقيقها
🎯 أمثلة عملية:
  • • التأكد من تنفيذ جميع الوظائف المطلوبة
  • • فحص أن النظام يتصرف كما هو مطلوب
  • • التحقق من معايير الأداء المحددة
  • • ضمان تحقيق متطلبات سهولة الاستخدام
⚖️

Verifying Compliance

التحقق من امتثال كائن الاختبار للمتطلبات التعاقدية والقانونية والتنظيمية
📜 أمثلة عملية:
  • • الامتثال لقوانين حماية البيانات (GDPR)
  • • التوافق مع معايير الصناعة (ISO, IEEE)
  • • الالتزام بالمتطلبات التعاقدية للعميل
  • • مطابقة اللوائح الحكومية والقطاعية
📢

Providing Information

توفير المعلومات لأصحاب المصلحة للسماح لهم باتخاذ قرارات مدروسة
📊 أمثلة عملية:
  • • تقارير جودة البرنامج للإدارة
  • • معلومات الجاهزية للإطلاق
  • • تقييم المخاطر المتبقية
  • • بيانات الأداء والاستقرار
💪

Building Confidence

بناء الثقة في جودة كائن الاختبار
🌟 أمثلة عملية:
  • • إثبات استقرار النظام تحت الضغط
  • • تأكيد موثوقية الوظائف الحرجة
  • • إظهار جودة تجربة المستخدم
  • • التحقق من أمان البيانات والنظام
🎭

Validating Completeness

التحقق من أن كائن الاختبار مكتمل ويعمل كما هو متوقع من قبل أصحاب المصلحة
🎪 أمثلة عملية:
  • • التأكد من اكتمال جميع الوظائف
  • • فحص تطابق التوقعات مع الواقع
  • • التحقق من سهولة الاستخدام للمستخدم النهائي
  • • ضمان تحقيق أهداف العمل

🌍 تأثير السياق على أهداف Testing:

📄 Work Product Type

نوع المنتج المختبر يحدد التركيز: تطبيق موبايل vs نظام طبي

🏗️ Test Level

مستوى الاختبار: Unit vs Integration vs System vs Acceptance

⚠️ Risks

مستوى المخاطر: عالي للأنظمة الحرجة، منخفض للتطبيقات البسيطة

🔄 SDLC Model

نموذج دورة التطوير: Waterfall vs Agile vs DevOps

🏢 Business Context

السياق التجاري: الهيكل المؤسسي، المنافسة، وقت الوصول للسوق

🔧 1.1.2 Testing and Debugging

الفرق بين Testing والـ Debugging

"Testing and debugging are separate activities. Testing can trigger failures that are caused by defects in the software (dynamic testing) or can directly find defects in the test object (static testing). When dynamic testing triggers a failure, debugging is concerned with finding causes of this failure (defects), analyzing these causes, and eliminating them."
Testing والـ Debugging أنشطة منفصلة. Testing يمكن أن يثير أعطال ناتجة عن عيوب في البرنامج (dynamic testing) أو يمكن أن يجد العيوب مباشرة في كائن الاختبار (static testing). عندما يثير Dynamic Testing عطل، الـ Debugging يهتم بإيجاد أسباب هذا العطل (العيوب)، تحليل هذه الأسباب، والقضاء عليها.

⚔️ مقارنة شاملة: Testing vs Debugging

🔍

Testing

التركيز: اكتشاف المشاكل
🎯 الأهداف:
  • اكتشاف وجود العيوب
  • التحقق من جودة البرنامج
  • إثبات أن النظام يعمل أو لا يعمل
  • تقييم مطابقة المتطلبات
⚙️ العملية:
  • تصميم وتنفيذ Test Cases
  • مقارنة النتائج الفعلية بالمتوقعة
  • توثيق العيوب المكتشفة
  • إنشاء تقارير الجودة
👥 المسؤولية:
  • QA Engineers
  • Test Analysts
  • End Users (في UAT)
  • Automated Test Tools
VS
🔧

Debugging

التركيز: إصلاح المشاكل
🎯 الأهداف:
  • تحديد السبب الجذري للعيب
  • إصلاح العيوب المكتشفة
  • منع تكرار نفس المشاكل
  • تحسين جودة الكود
⚙️ العملية:
  • إعادة إنتاج العطل
  • تحليل وتشخيص السبب
  • إصلاح الكود
  • اختبار الإصلاح (Confirmation Testing)
👥 المسؤولية:
  • Software Developers
  • System Engineers
  • Technical Leads
  • Code Authors

🔄 عملية Debugging التفصيلية:

1. Reproduction
إعادة إنتاج العطل
→
2. Diagnosis
تشخيص السبب الجذري
→
3. Fixing
إصلاح السبب
→
4. Confirmation
اختبار الإصلاح
→
5. Regression
اختبار عدم كسر شيء آخر

💼 مثال عملي من الواقع:

🔍 Testing Phase:

"المختبر يجرب تسجيل الدخول بكلمة مرور صحيحة، لكن النظام يرفض الدخول ويظهر رسالة خطأ. يقوم بتوثيق هذا العيب مع الخطوات التفصيلية."

→
🔧 Debugging Phase:

"المطور يستخدم الـ debugger لتتبع الكود، يكتشف أن المقارنة case-sensitive بالخطأ. يصلح الكود ويختبر الإصلاح."

📊 الفروقات الأساسية:

🎯 الغرض

Testing: إيجاد المشاكل
Debugging: حل المشاكل

⏰ التوقيت

Testing: بعد كتابة الكود
Debugging: بعد اكتشاف العيوب

🧠 المعرفة المطلوبة

Testing: فهم المتطلبات
Debugging: فهم الكود

🤖 الأتمتة

Testing: يمكن أتمتته
Debugging: دائماً يدوي

📊 النتيجة

Testing: تقرير بالعيوب
Debugging: كود مُصحح

🏗️ النهج

Testing: منهجي ومخطط
Debugging: استقصائي وتحليلي

💡 نقاط مهمة للتذكر:

  • أنشطة منفصلة: Testing يكتشف، Debugging يصلح
  • أدوار مختلفة: Testers vs Developers
  • مهارات مختلفة: اختبار المتطلبات vs فهم الكود
  • توقيت مختلف: Testing مستمر، Debugging عند الحاجة
  • تكامل ضروري: الاثنان يحتاجان بعض لضمان الجودة
  • هدف مشترك: تحسين جودة البرنامج للمستخدم النهائي
