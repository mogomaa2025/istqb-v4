⚙️ 1.4.1 Test Activities and Tasks

أنشطة ومهام Testing

"A test process usually consists of the main groups of activities described below. Although many of these activities may appear to follow a logical sequence, they are often implemented iteratively or in parallel."
عملية الاختبار عادة بتتكون من المجموعات الرئيسية من الأنشطة المذكورة أدناه. رغم إن الأنشطة دي ممكن تبدو إنها بتتبع تسلسل منطقي، إلا إنها غالباً بتتنفذ بشكل تكراري أو متوازي.
📋

Test Planning

"إيه هدف Testing؟"
تحديد أهداف الاختبار واختيار النهج الأنسب لتحقيق هذه الأهداف في حدود القيود المفروضة
🎯 أمثلة عملية:
  • • تحديد نطاق الاختبار والأهداف
  • • اختيار استراتيجية الاختبار
  • • تقدير الوقت والموارد
  • • تحديد معايير البدء والانتهاء
📊

Test Monitoring & Control

"إزاي سير العمل؟"
مراقبة مستمرة لجميع أنشطة الاختبار ومقارنة التقدم الفعلي مع الخطة واتخاذ الإجراءات اللازمة
📈 أمثلة عملية:
  • • تقارير التقدم اليومية
  • • مراقبة معايير الجودة
  • • تعديل الخطط حسب الحاجة
  • • إدارة المخاطر والمشاكل
🔍

Test Analysis

"إيه اللي نختبره؟"
تحليل أساس الاختبار لتحديد الخصائص القابلة للاختبار وتعريف وترتيب أولوية شروط الاختبار
🔬 أمثلة عملية:
  • • تحليل المتطلبات والمواصفات
  • • تحديد المخاطر ومستوياتها
  • • إنشاء معايير التغطية
  • • تقييم قابلية الاختبار
🎨

Test Design

"إزاي نختبر؟"
تطوير شروط الاختبار إلى حالات اختبار ومنتجات عمل أخرى مثل تصميم البيئة وتحديد البيانات
🎯 أمثلة عملية:
  • • تصميم حالات الاختبار
  • • تحديد بيانات الاختبار
  • • تصميم بيئة الاختبار
  • • إنشاء Test Charters للاختبار الاستكشافي
🔧

Test Implementation

"إزاي نحضر للتنفيذ؟"
إنشاء أو الحصول على منتجات العمل اللازمة لتنفيذ الاختبار مثل البيانات والسكريبتات والبيئة
🛠️ أمثلة عملية:
  • • إنشاء إجراءات الاختبار
  • • كتابة السكريبتات اليدوية والتلقائية
  • • تحضير بيانات الاختبار
  • • إعداد بيئة الاختبار
▶️

Test Execution

"إزاي ننفذ الاختبارات؟"
تشغيل الاختبارات وفقاً لجدولة التنفيذ ومقارنة النتائج الفعلية بالمتوقعة وتسجيل النتائج
🎬 أمثلة عملية:
  • • تنفيذ حالات الاختبار
  • • مقارنة النتائج الفعلية بالمتوقعة
  • • تسجيل نتائج الاختبار
  • • تحليل الشذوذ وإبلاغ العيوب
🏁

Test Completion

"إيه اللي اتعلمناه؟"
أنشطة إنهاء الاختبار وإنشاء التقارير النهائية والدروس المستفادة وتسليم المنتجات
📋 أمثلة عملية:
  • • تقرير إنهاء الاختبار
  • • أرشفة منتجات العمل المفيدة
  • • الدروس المستفادة للمستقبل
  • • إغلاق بيئة الاختبار

🌍 1.4.2 Test Process in Context

عملية Testing في السياق

"Testing is not performed in isolation. Test activities are an integral part of the development processes carried out within an organization. Testing is also funded by stakeholders and its final goal is to help fulfill the stakeholders' business needs."
Testing مابيتمش بمعزل عن باقي العمليات. أنشطة الاختبار هي جزء لا يتجزأ من عمليات التطوير في المنظمة. Testing كمان ممول من أصحاب المصلحة وهدفه النهائي مساعدة في تحقيق احتياجات أصحاب المصلحة التجارية.

🎯 العوامل المؤثرة على عملية Testing:

👥
Stakeholders

الاحتياجات، التوقعات، المتطلبات، الرغبة في التعاون، الميزانية المتاحة

🎓
Team Members

المهارات، المعرفة، مستوى الخبرة، التوافر، احتياجات التدريب

🏢
Business Domain

حرجية النظام، المخاطر المحددة، احتياجات السوق، اللوائح القانونية

⚙️
Technical Factors

نوع البرمجيات، هيكل المنتج، التقنيات المستخدمة، التعقيد التقني

📅
Project Constraints

النطاق، الوقت، الميزانية، الموارد المتاحة، المواعيد النهائية

🏛️
Organizational Factors

الهيكل التنظيمي، السياسات الموجودة، الممارسات المتبعة، الثقافة

🔄
SDLC

ممارسات الهندسة، طرق التطوير، نموذج دورة الحياة المتبع

🛠️
Tools

التوافر، سهولة الاستخدام، الامتثال للمعايير، التكلفة

🎯 تأثير العوامل على قرارات Testing:

📋 استراتيجية الاختبار

مشروع طبي = تركيز على الأمان
لعبة = تركيز على الأداء والمتعة

🔧 تقنيات الاختبار

فريق خبير = تقنيات متقدمة
فريق مبتدئ = تقنيات بسيطة

🤖 درجة الأتمتة

مشروع طويل المدى = أتمتة عالية
مشروع قصير = اختبار يدوي

📊 مستوى التغطية

نظام حساس = تغطية 100%
تطبيق بسيط = تغطية 70%

📄 تفاصيل التوثيق

صناعة منظمة = توثيق مفصل
Startup = توثيق مختصر

📈 التقارير

فريق موزع = تقارير تفصيلية
فريق واحد = تقارير بسيطة

💡 الدروس الرئيسية:

  • مافيش نهج واحد يناسب الكل: كل مشروع له ظروفه الخاصة
  • التخصيص ضروري: عملية Testing لازم تتكيف مع السياق
  • التوازن مطلوب: بين الجودة والوقت والتكلفة
  • التواصل مهم: فهم احتياجات أصحاب المصلحة
  • المرونة أساسية: التكيف مع التغييرات والتطورات

📄 1.4.3 Testware

منتجات عمل Testing

"Testware is created as output work products from the test activities described in section 1.4.1. There is a significant variation in how different organizations produce, shape, name, organize and manage their work products."
Testware بيتم إنشاؤه كـمنتجات عمل مخرجة من أنشطة الاختبار المذكورة في القسم 1.4.1. فيه تنوع كبير في كيفية إنتاج وتشكيل وتسمية وتنظيم وإدارة منتجات العمل دي في المنظمات المختلفة.

📋 منتجات العمل حسب النشاط:

📋 Test Planning منتجات
  • Test Plan (خطة الاختبار الشاملة)
  • Test Schedule (جدولة الاختبارات)
  • Risk Register (سجل المخاطر)
  • Entry & Exit Criteria (معايير البدء والانتهاء)
📊 Test Monitoring منتجات
  • Test Progress Reports (تقارير التقدم)
  • Control Directives (توجيهات التحكم)
  • Risk Information (معلومات المخاطر)
  • Status Updates (تحديثات الوضع)
🔍 Test Analysis منتجات
  • Test Conditions (شروط الاختبار)
  • Acceptance Criteria (معايير القبول)
  • Defect Reports (تقارير العيوب في الأساس)
  • Testability Assessment (تقييم قابلية الاختبار)
🎨 Test Design منتجات
  • Test Cases (حالات الاختبار)
  • Test Charters (مواثيق الاختبار)
  • Coverage Items (عناصر التغطية)
  • Test Data Requirements (متطلبات البيانات)
  • Test Environment Requirements (متطلبات البيئة)
🔧 Test Implementation منتجات
  • Test Procedures (إجراءات الاختبار)
  • Automated Test Scripts (السكريبتات التلقائية)
  • Test Suites (مجموعات الاختبار)
  • Test Data (بيانات الاختبار)
  • Test Execution Schedule (جدولة التنفيذ)
  • Test Environment Elements (عناصر البيئة)
▶️ Test Execution منتجات
  • Test Logs (سجلات الاختبار)
  • Defect Reports (تقارير العيوب المكتشفة)
  • Actual Results (النتائج الفعلية)
  • Test Evidence (أدلة الاختبار)
🏁 Test Completion منتجات
  • Test Completion Report (تقرير إنهاء الاختبار)
  • Action Items (عناصر العمل للمستقبل)
  • Lessons Learned (الدروس المستفادة)
  • Change Requests (طلبات التغيير)

🌐 عناصر بيئة الاختبار:

🔌 Stubs

مكونات وهمية للأنظمة التي لم تكتمل بعد

🚗 Drivers

برامج تحكم لتشغيل المكونات المراد اختبارها

🎮 Simulators

محاكيات للأجهزة أو البيئات الحقيقية

☁️ Service Virtualizations

محاكاة الخدمات الخارجية والـ APIs

📊 أهمية إدارة Testware:

  • Configuration Management: ضمان الاتساق وسلامة منتجات العمل
  • إعادة الاستخدام: استخدام منتجات العمل في مشاريع أخرى
  • الأرشفة: حفظ المنتجات المفيدة للمستقبل
  • التسليم: نقل المنتجات للفرق المناسبة
  • التوثيق: توفير مرجع للمراجعات والتدقيق

🔗 1.4.4 Traceability between Test Basis and Testware

الربط بين أساس الاختبار ومنتجات العمل

"In order to implement effective test monitoring and control, it is important to establish and maintain traceability throughout the test process between the test basis elements, testware associated with these elements, test results, and detected defects."
لتنفيذ مراقبة وتحكم فعالين في الاختبار، من المهم إنشاء والمحافظة على الربط عبر عملية الاختبار بين عناصر أساس الاختبار، منتجات العمل المرتبطة بها، نتائج الاختبار، والعيوب المكتشفة.

🔄 سلسلة الربط (Traceability Chain):

📋 Requirements

متطلبات النظام والوظائف المطلوبة

→
🎯 Test Conditions

شروط الاختبار المشتقة من المتطلبات

→
🧪 Test Cases

حالات الاختبار التفصيلية

→
📊 Test Results

نتائج تنفيذ الاختبارات

→
🐛 Defects

العيوب المكتشفة والمرتبطة بالمتطلبات

🎯 فوائد Traceability:

📊 Coverage Evaluation

المثال: تتبع حالات الاختبار للمتطلبات للتأكد من تغطية جميع المتطلبات

⚠️ Risk Assessment

المثال: ربط نتائج الاختبار بالمخاطر لتقييم مستوى المخاطر المتبقية

🔄 Impact Analysis

المثال: عند تغيير متطلب، معرفة أي اختبارات تتأثر

🔍 Test Audits

المثال: تسهيل مراجعة العمليات ومطابقة معايير الحوكمة

📈 Progress Reports

المثال: تقارير واضحة تتضمن حالة عناصر أساس الاختبار

💬 Communication

المثال: شرح الجوانب التقنية لأصحاب المصلحة بطريقة مفهومة

💼 أمثلة عملية على Traceability:

🏦 نظام بنكي:

المتطلب: "يجب أن يدعم النظام تحويل الأموال بين الحسابات"

شرط الاختبار: اختبار تحويل الأموال في سيناريوهات مختلفة

حالة الاختبار: تحويل 1000 جنيه من حساب A إلى حساب B

النتيجة: تم التحويل بنجاح وتحديث الأرصدة

التغطية: المتطلب مغطى 100%

🛒 متجر إلكتروني:

المخاطرة: فقدان بيانات الطلبات (مخاطرة عالية)

اختبار المخاطرة: اختبار حفظ الطلبات في حالات الأعطال

النتيجة: تم اكتشاف مشكلة في حفظ البيانات

تقييم المخاطرة: المخاطرة المتبقية: متوسطة بعد الإصلاح

🔑 مفاتيح نجاح Traceability:

  • معايير التغطية القابلة للقياس: تحديد KPIs واضحة
  • أدوات مناسبة: استخدام أدوات تدعم الربط التلقائي
  • التحديث المستمر: المحافظة على الروابط عند التغييرات
  • الوضوح والبساطة: تقارير مفهومة لجميع الأطراف
  • التدريب: تأهيل الفريق على أهمية وطرق الربط

👥 1.4.5 Roles in Testing

الأدوار في Testing

"In this syllabus, two principal roles in testing are covered: a test management role and a testing role. The activities and tasks assigned to these two roles depend on factors such as the project and product context, the skills of the people in the roles, and the organization."
في هذا المنهج، يتم تغطية دورين رئيسيين في Testing: دور إدارة الاختبار و دور الاختبار. الأنشطة والمهام المخصصة لهذين الدورين تعتمد على عوامل مثل سياق المشروع والمنتج، مهارات الأشخاص في الأدوار، والمنظمة.
👔

Test Management Role

التركيز: الإدارة والقيادة
🎯 الأنشطة الرئيسية:
  • Test Planning (تخطيط الاختبار)
  • Test Monitoring & Control (مراقبة وتحكم)
  • Test Completion (إنهاء الاختبار)
  • Resource Management (إدارة الموارد)
  • Stakeholder Communication (التواصل)
👥 أمثلة الأدوار:
  • • Test Manager
  • • Team Leader
  • • Development Manager
  • • Project Manager (في بعض الحالات)
VS
🔬

Testing Role

التركيز: الجوانب التقنية
🛠️ الأنشطة الرئيسية:
  • Test Analysis (تحليل الاختبار)
  • Test Design (تصميم الاختبار)
  • Test Implementation (تنفيذ الاختبار)
  • Test Execution (تشغيل الاختبار)
  • Technical Documentation (التوثيق التقني)
👨‍💻 أمثلة الأدوار:
  • • Tester / QA Engineer
  • • Test Analyst
  • • Test Automation Engineer
  • • Developer (في Developer Testing)

🔄 تنوع الأدوار حسب السياق:

🏃‍♂️ Agile Teams

التغيير:

  • • بعض مهام Test Management يقوم بها الفريق
  • • Product Owner يشارك في التخطيط
  • • Scrum Master يدير العملية
  • • الأدوار أكثر مرونة وتداخلاً
🏢 Enterprise Projects

التغيير:

  • • أدوار محددة ومنفصلة
  • • Test Managers متخصصون
  • • فرق اختبار مستقلة
  • • هيكل تنظيمي واضح
👨‍💻 Small Teams

التغيير:

  • • شخص واحد يقوم بدورين
  • • أدوار متداخلة ومرنة
  • • تبادل المسؤوليات
  • • تركيز على النتائج أكثر من العمليات
🌐 Distributed Teams

التغيير:

  • • Test Manager في كل موقع
  • • تنسيق بين المواقع المختلفة
  • • أدوات تواصل وإدارة متقدمة
  • • معايير موحدة عبر المواقع

🔄 مرونة الأدوار:

⏰ Same Person, Different Times

نفس الشخص يقوم بأدوار مختلفة في أوقات مختلفة من المشروع

🔀 Role Switching

الانتقال بين دور الإدارة والدور التقني حسب احتياجات المشروع

👥 Team Collaboration

تشارك المسؤوليات والتعاون بين أعضاء الفريق

💡 مبادئ نجاح الأدوار:

  • الوضوح: تحديد واضح للمسؤوليات والصلاحيات
  • المرونة: التكيف مع احتياجات المشروع والمنظمة
  • التواصل: تنسيق فعال بين الأدوار المختلفة
  • التطوير: بناء المهارات اللازمة لكل دور
  • التوازن: توزيع عادل للأعباء والمسؤوليات
